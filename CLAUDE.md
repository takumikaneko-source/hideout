クロスプロジェクト知識庫 hideout での Claude 利用時の指針・プリファレンス。

## 1. 基本方針

### 言語・口調
- **日本語のみ**、丁寧でフォーマルな文体
- 結論を先に述べ、理由・詳細は後から
- 過度な前置きを避け、要点に集中

### 出力形式
- 説明は文章中心。箇条書きは整理に有効な場合のみ使用
- 資料作成時は **Markdown が基本**
  - 図や表が必要な場合は、作成前に形式を確認（Markdown 表 / 図 / HTML 等）
  - 閲覧用・説明用資料は **HTML** で作成（単一ファイル・外部依存なし）
  - HTML資料の体裁: 章は番号付き（1. 2. …）、小単位は連番（3-1, 3-2）、右追従型目次付き

### 文章・資料のルール
- **AI臭い言い回し禁止**
  - 避ける例: 「中核を実証」「一気通貫」「技術の核」「堅牢」「最優先」「差し替え可能な部品」
  - 使う例: 「ひと通り動く」「動作確認」「重要な点」「重視」「発生源ごとに切り替え」
- **個人特定情報をぼかす**
  - 避ける: 「上長」「○○さん」「KDDI 林さん」等
  - 使う: 「案として挙がった」「運用チーム」「関係者」等
- **機密情報は含めない**
  - AWS認証情報・パスワード・秘密鍵・個人認証情報を絶対に含めない
  - 作成する資料・ドキュメントに機密情報が含まれそうな場合は、作成前に明示的に確認

## 2. コーディング方針

### 既存スタイルの尊重
- 既存コードの命名規則・依存関係・構造を勝手に変更しない
- 変更の意図と影響範囲を簡潔に説明する
- 破壊的な変更（削除・大規模リファクタ・依存追加）は事前に確認する

### コメント・ドキュメント
- コメントは過剰にせず、非自明な箇所のみ簡潔に付す
- WHAT ではなく WHY を記す（WHAT は well-named identifiers が担当）
- 現在の issue/PR/タスク に言及しない（それは git log の役割）

### テスト・動作確認
- 破壊的でない変更でも、実装後は動作確認の観点を添える
- テストはタイプチェック・lint 完了後、実際に機能を駆動して検証

---

## 3. Git / コミット管理

### コミット・PR・Push
- コミット・PR・push は、**ユーザーが明示的に指示したときのみ** 実行
- コミットメッセージは **日本語** で記述
- 機能単位の変更は `【機能名】内容` 形式（例: `【マート処理】起動時間変更対応`）
- コミット時にレビュー対象外のファイル（`.env`, 認証情報）が含まれていないか確認

### 破壊的操作
- `git reset --hard`, `git checkout --`, `git push --force` などは本当に必要な場合のみ
- git status を実行して、意図しない変更がないことを確認してから操作

## 4. hideout リポジトリ構成と利用方法

### ディレクトリ構成
```
hideout/
├── 作業履歴/                        # プロジェクト実施履歴（日付フォルダ）
│   ├── YYYYMMDD_プロジェクト名/
│   │   ├── README.md （作業内容記録）
│   │   ├── 設計資料/
│   │   └── 成果物/
│   ├── 20260709_hideout初期化/
│   ├── 20260629_人間AWS自動化/
│   └── ...
│
├── 参考資料/                        # 設計資料・仕様書・参照資料
│   ├── 矢崎案件/
│   └── ...
│
├── テンプレート/                    # 共通資産・テンプレート
│   ├── アイコン/
│   ├── メモ.*
│   └── ...
│
├── 知識ベース/                      # 横断的ナレッジ
│   ├── sql/
│   └── ...
│
└── README.md
```

### 作業開始時の流れ
1. 新規作業・プロジェクト開始時: `作業履歴/YYYYMMDD_プロジェクト名/` フォルダを作成
2. `テンプレート/作業記録.md` をコピーして `README.md` とし、背景・目的を記入
3. 設計・実装・テスト結果を該当フォルダに格納し、README.md に経過を追記
4. 完了時に結果・残課題を記入（commit・push はユーザーの指示があったときのみ）

### 資料配置ルール
- **参照資料・設計書（受領した原本）** → `参考資料/`
- **テンプレート・共通部品** → `テンプレート/`
- **実装履歴・成果物・試行錯誤記録** → `作業履歴/YYYYMMDD_*/`
- **共有ナレッジ・教訓・FAQ** → `知識ベース/`

各フォルダ直下の `README.md` に詳細な配置ルールと索引がある。迷ったらまずそれを読む。

### テンプレートの使用（Claude は資料作成時にここを最初に確認する）
| 作るもの | 使うテンプレート |
|----------|------------------|
| 作業履歴の README.md | `テンプレート/作業記録.md` |
| 知識ベースの記事 | `テンプレート/ナレッジ記事.md` |
| 手順書 | `テンプレート/手順書.md` |
| 閲覧用HTML資料 | `テンプレート/HTML資料スケルトン.html`（右追従目次・番号付き見出し実装済み） |

### 知識の振り分けルール（Claude の作業終了時フロー）
作業がひと区切りついたら、Claude は次を確認・実行する:

1. **作業記録の更新**: 当該 `作業履歴/YYYYMMDD_*/README.md` に結果・決定事項を反映したか
2. **ナレッジの切り出し**: 作業中に得た「他でも使い回せる知見」（AWSの挙動、SQLパターン、組織の制約、ハマりどころ等）があれば、`テンプレート/ナレッジ記事.md` を使って `知識ベース/` に記事化し、`知識ベース/README.md` の索引に1行追記する
3. **リンクの相互参照**: 作業記録 → ナレッジ記事、ナレッジ記事 → 出典の作業記録、を相互リンクする
4. **索引の更新漏れ確認**: 各フォルダの README.md の索引・一覧が実態とずれていないか

このフローは Claude が自発的に提案してよい（「この知見は知識ベースに切り出しますか？」）。
ただしファイルの作成・移動はユーザーへの報告を伴い、commit・push は指示があったときのみ。

## 7. プリファレンス・スタイル

### コード品質
- 既存コードスタイル・命名規則を尊重
- 変更の必要性と影響範囲を明示
- コメントは非自明な箇所のみ

### タスク管理
- 前置きせず、すぐに着手する
- 不確実な点は推測と明示し、事実と区別する
- 誤りに気づいたら率直に訂正

### 資料作成
- 形式の指定がなければ Markdown か HTML を選択
- 機密情報・個人情報は除外
- AI臭い表現・大げさな言い回しを避ける

## 9. よくある質問・注意事項

### Q. 機密情報が混在した資料を作成する場合は？
**A.** 作成前に明示的に確認します。AWS認証情報・パスワード・個人認証情報は絶対に含めません。

### Q. 既存のコードスタイルが不明な場合は？
**A.** 既存ファイルを読み込んで確認、または実装前に確認をとります。

### Q. プロジェクト間で矛盾する方針がある場合は？
**A.** hideout > taxi-cloud > グローバル設定 の優先順で判断し、必要に応じてユーザーに確認します。
