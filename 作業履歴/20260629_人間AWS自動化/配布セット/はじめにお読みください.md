# アラート対応 半自動化ツール 一式

CloudWatch のエラーアラートについて、**①ログの自動収集** と **②対応要否・影響度のAI判定** を行うツールをまとめた配布セットです。最終判断は運用チームの目視で行います（AIは参考情報）。

---

## フォルダ構成

作業者が触るのは、各フォルダ直下にある **実行用の .bat** と **settings.py**、そして **INPUT / OUTPUT** だけです。本体プログラムは各 `program/` にまとめてあり、触る必要はありません。

```
配布セット/
├─ はじめにお読みください.md   ← この資料
├─ 環境チェック.bat            ← まず実行。必要なものが揃っているか点検（AI/AWS不使用・安全）
├─ セットアップ.bat            ← 両ツールの依存パッケージをまとめて導入
├─ 資料/                       ← 参考資材（全体仕様・使い方の説明HTML）
├─ program/                    ← 触らない（環境チェックの本体 env_check.py）
├─ 1_ログ収集ツール/           ← ①CloudWatchからエラーログを収集し、Excelへ転記
│   ├─ 収集実行.bat           ← ★ダブルクリックで起動
│   ├─ settings.py             ← ★設定はここだけ編集
│   ├─ INPUT/                  （ダウンロードした .xlsx をここに置く）
│   ├─ OUTPUT/                 （収集結果 …_エラーログ収集後.xlsx が出る）
│   └─ program/                ← 触らない（collect_logs.py / requirements / 手順書.html）
└─ 2_判定ツール/               ← ②収集済みログをAIが判定し、Excelへ書き込み
    ├─ 判定実行.bat           ← ★ダブルクリックで起動
    ├─ settings.py             ← ★設定はここだけ編集
    ├─ ナレッジ/               （判定基準・過去事例。正本はGoogleドライブ ※下記参照。同梱分は初期スナップショット）
    ├─ OUTPUT/                 （判定結果 …_判定済.xlsx と、差分記録 AI判定_運用差分記録.xlsx が出る）
    └─ program/                ← 触らない（judge_logs.py / requirements / README.md）
```

参考資材は `資料/` にあります（全体の仕様・使い方の説明HTML）。判定ツールの詳細は `2_判定ツール/program/README.md`、収集ツールの手順は `1_ログ収集ツール/program/手順書.html` を参照してください。

- 各ツールは **フォルダ直下の実行 .bat（`収集実行.bat` / `判定実行.bat`）をダブルクリック** すれば動きます（画面はすぐ閉じません）。
- 設定は各フォルダ直下の **`settings.py`** だけを編集します。`program/` の中身は変更不要です。
- 使い方だけ見たいときは、実行 .bat に `--help` を付けて実行するとヘルプが出ます。
- 相対パスはツールのフォルダ基準で解決されるので、この「配布セット」フォルダごと別の場所へ移動しても動きます。

## 使い方（全体の流れ）

### 事前準備（初回のみ）

1. **Python 3.10 以上**を用意する（[python.org](https://www.python.org/) 版を推奨。Microsoft Store 版は不具合が出ることがあります → 下の「困ったとき」参照）。
2. **`セットアップ.bat`** をダブルクリックして、両ツールの依存パッケージをまとめて導入する。
   （うまくいかない場合は各フォルダで `pip install -r requirements.txt`）
3. 判定ツールをAIサブスク（Claude Code）で動かす場合は、`claude` を導入して一度ログインしておく。
   ```
   npm install -g @anthropic-ai/claude-code
   claude
   ```
   （APIキーは不要。従量課金のAPIを使う場合のみ `2_判定ツール/settings.py` の `BACKEND="api"` と環境変数 `ANTHROPIC_API_KEY` を設定）
4. **`環境チェック.bat`** をダブルクリックして、必要なものが揃っているか点検する（AI/AWSは呼ばない安全な確認）。「すべて良好」になれば準備完了です。

### 入力の用意

判定の入力は「**エラーログ転記済みの xlsx**」です。収集ツール（`1_ログ収集ツール`）で用意したその出力ファイルを、既定の入力先 `1_ログ収集ツール/OUTPUT/` に置いておきます（収集ツールの操作手順はこの資料では割愛します。詳細は `1_ログ収集ツール/program/手順書.html`）。別の場所にある場合は `2_判定ツール/settings.py` の `INPUT_XLSX` をそのパスに変更します。

### AI判定の実行

1. `2_判定ツール/settings.py` を開き、`INPUT_XLSX` が入力ファイルを指しているか確認する（既定でOK）。
2. まずは `MAX_ROWS = 5` など少数で試す。
3. `2_判定ツール/判定実行.bat` をダブルクリックして実行する。
   - 対象は「D列にログがあり、G列が空」の行。
   - 結果は `OUTPUT/…_判定済.xlsx` の G〜K列（AI対応要否・影響度・理由・確信度・判定方式）に出力。
   - 詳細は `2_判定ツール/program/README.md` を参照。

### 運用チームの最終判断と差分の記録

- 判定済みファイルの **E列（対応要不要）** に、運用チームが最終判断を記入する。
- AI判定と運用判断の差分は、判定ツールが自動で **`OUTPUT/AI判定_運用差分記録.xlsx`** に蓄積する（判定済みファイルは毎回作り直されるため、差分は別ファイルに残す）。
  - 「運用対応要否」「差分理由・改善メモ」列に運用チームが記入すると、「差分」列が自動で「一致 / 差分」を表示する。
  - この差分記録を見直して、`ナレッジ/` の判定基準や事例を継続的に改善していく。

## ナレッジの正本（Google ドライブ）

判定に使うナレッジ（`rules.yaml` / `cases.jsonl` / `guidelines.md` / `system_context.md`）の**正本は次の Google ドライブ共有フォルダ**です。最新の判定基準・事例はここで一元管理します。

- https://drive.google.com/drive/folders/1imP27rOCra7HNo2jAHYeaG4rmineMVVp

判定ツールはオフラインで動く Python なので、実行時にネット越しで直接ここを読むことはできません。各PCでは、次のどちらかで**ローカルに用意した最新版**を参照させます（`2_判定ツール/settings.py` の `KNOWLEDGE_DIR` で場所を指定）。

- **(A) 同期して参照（推奨）**: 「Google ドライブ デスクトップ」を入れ、上記フォルダを自分のドライブに追加してローカル同期し、その同期パスを `KNOWLEDGE_DIR` に指定する。以降は常に最新を読めます。
  - **初期設定時に作業者の操作が必要です**（Google アカウントでのサインインと、共有フォルダの追加・同期の許可）。社内ポリシーによっては情報システム部門の承認が要る場合があります。
- **(B) 手動ダウンロード**: 上記フォルダから4ファイルをダウンロードし、`2_判定ツール/ナレッジ/` に上書きする。手軽ですが、更新のたびに入れ替えが必要です。

同梱の `2_判定ツール/ナレッジ/` は**初期スナップショット（フォールバック）**です。Drive を参照できない環境でもとりあえず動きますが、内容は配布時点のものです。

## 困ったとき（トラブルシュート）

まず **`環境チェック.bat`** を実行すると、下の多くを自動で検出し、対処を表示します。

| 症状 | 原因 | 対処 |
|------|------|------|
| `claude … が見つかりませんでした` | Claude CLI が未導入 | `npm install -g @anthropic-ai/claude-code` → `claude` でログイン |
| npm で `SELF_SIGNED_CERT_IN_CHAIN` | 社内のTLS検査プロキシ | Windowsの証明書を書き出して npm に渡す（下の「社内証明書」参照） |
| `claude.cmd` を「なし」と誤検出 | **Microsoft Store 版 Python** で AppData が見えない | python.org 版か pyenv 版（実体）を使う。`実行.bat` は実体を自動選択します |
| `.bat` で `'cp' は認識されません` 等 | `.bat` の文字コード/改行の問題 | 同梱の `.bat` は ASCII・CRLF 済み。自作する場合は同様にする |
| パッケージ未導入エラー | 依存が未インストール | `セットアップ.bat`、または各フォルダで `pip install -r requirements.txt` |
| 保存/書き込みに失敗 | 対象 Excel を開いたまま | その Excel を閉じてから再実行 |
| 処理される行が少ない（収集） | 対象は「A・Bがあり **D列が空**」の行だけ | 再取得したい行は D列を空にする |
| `使用上限` で未判定（判定） | サブスクの使用枠上限 | 時間をおいて再実行。`MAX_ROWS` を小さくして回す |

### 社内証明書（プロキシでSSLエラーが出る場合）

Windowsの信頼済み証明書を1ファイルに書き出し、npm と（必要なら）収集ツールに渡します（PowerShellで実行）。

```powershell
$out = "$env:USERPROFILE\corp-ca-bundle.pem"
$certs = Get-ChildItem Cert:\LocalMachine\Root, Cert:\CurrentUser\Root
$lines = foreach ($c in $certs) { "-----BEGIN CERTIFICATE-----"; [Convert]::ToBase64String($c.RawData,'InsertLineBreaks'); "-----END CERTIFICATE-----" }
$lines | Set-Content -Encoding ascii $out
npm config set cafile "$out"     # npm に社内CAを渡す
```

作成した `corp-ca-bundle.pem` は、収集ツールの `settings.py` の `CA_BUNDLE` にも指定できます。

## 注意事項

- **AWSの資格情報（アクセスキー・MFA等）はAIに渡しません。** ログ収集はAIを使わず、AWSへの接続は運用担当者の手元で行います。
- 判定ツールが使うのは「収集済みのログ本文」だけです。
- Excelでファイルを開いたままだと書き込みに失敗します。実行時は対象ファイルを閉じてください。
- **AIの判定は参考情報**です。最終判断は必ず運用チームの目視で行ってください。
