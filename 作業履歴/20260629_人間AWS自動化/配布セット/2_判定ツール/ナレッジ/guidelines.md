# 判定基準（guidelines）

AIが「影響度」と「対応要否」を判定する際に参照する基準。
出典: `../資料/運用作業手順.xlsx`「暫定 日次監視内容・手順」「備忘：ERROR対応例、lambdaの設定確認方法」を整理。個人名・顧客コード・内部URLは判定に不要なため除いている。

判定の native な軸は「対応要否（静観／対応要／不要）」。影響度（大/中/小）はその補助として付す。

## 0. 全体の原則

- **静観としてよいもの**
  - ユーザー側の再操作で解消可能なもの（WebAPI・画面側でエラーを表示しているケース）
  - 商用環境への反映が必要でないもの
  - リトライ・再処理で最終的に解消しているもの
- **必ず確認すること**
  - リトライが終わってもエラーが出続けていないか
  - エラーが異常に多く出力されていないか
  - **エラー箇所の前後関係**: 上流起因のサイン（S3 404・入力データ不正・並列アラームの同時発報）がないか、下流へ波及するか（処理チェーンの中間か終端か・次サイクルで自然回復するか）。判断材料は `system_context.md` の3章「処理フローと影響伝播」・4章「一般則」を参照
- **顧客への問い合わせ**は、内部（管理者 PM／PL）・関係部署の確認をしてから行う。データの要不要をこちら側だけで判断しない。
- **自動復旧基盤により Backlog 起票**されたものは緊急度高めのため、優先して確認する。

## 1. 影響度の定義（補助軸）

| 影響度 | 目安 |
|--------|------|
| 大 | 業務停止・データ欠損が残る／高額請求など波及の恐れ／復旧不可 |
| 中 | 単発の失敗。リトライ等で解消見込みだが、データ整合の確認が要る |
| 小 | 自動リトライ・再処理・正常終了済みで解消、業務影響なし |

## 2. 対応要否の定義

| 対応要否 | 意味 |
|----------|------|
| 不要 | 既知・自動解消。対応不要 |
| 静観 | 今回は対応不要だが、再発したら確認する |
| 要 | 人手での調査・再取り込み・顧客照会・エスカレーションが必要 |

## 3. ケース別の判定

### 3-1. データ取り込み・ファイル処理エラー

**A. 処理済みデータ／ファイル未格納**
すでに正常終了しているデータの再取り込み、または対象ファイルが見つからないケース。
- 確認: エラー出力より前のログを遡り、該当データの取り込みが「正常終了」しているか（バケット名・エンティティ名まで確認）。
- 判定: エラー前に取り込み正常終了 → **静観（小）**。取り込まれていない想定外エラー → **要（起票・開発調査依頼）**。

**B. データ取り込み時のエラー**
- ケース1（リトライ設定あり）: 再実行予定時刻に処理が走り、DBに取り込み済みか／ファイルが所定の場所へ移動済みか／SQSにキュー滞留がないかを確認。取り込み成功 → **静観**。滞留や未取り込みが残る → **要（商用再取り込み調整）**。
- ケース2（仕様外ファイル）: 形式誤り・必須項目欠落・テストデータ混入など → **要（顧客へ正しいファイルの再連携を依頼）**。ただしテスト乗務員コード `999999` は取り込み不要。

**C. 仕様外データ起因**
- 車両マスタ ICCID 重複（ログに `MultipleResultsFound`）: リトライで取り込み成功 → **静観可**。生きレコードが2件以上／取り込み失敗が残る／同一車両で同一エラー再発 → **要（顧客へ車両コード登録状況を照会）**。
- 連携データ欠損（ログに「選択された目標設定のデータがマスタに存在しません」等）: 改修済みの既知事象。必要なマスタは再取り込み。

### 3-2. システムリソース・実行環境のエラー

**A. メモリ不足（OOM）**
- 検知: `Runtime exited with error: signal: killed` / `OutOfMemory`。REPORT行の Max Memory Used が Memory Size とほぼ同値でログが途中で止まる。
- 判定: データ取りこぼしがなければ **静観寄り（中）**。取りこぼしあり → **要（再取り込み調整）**。

**B. タイムアウト**
- 検知: `timeout`。REPORT行の Duration が設定上限に到達。直前ログが「DB接続開始」「外部APIリクエスト」で止まっていれば接続先待ちの可能性。
- 判定: リトライで最終的に成功 → **静観**。取りこぼしが残る → **要**。

### 3-3. Step Functions

- **A. タイムアウト・遅延**: リトライ設定で最終的に成功しているか確認。成功 → 静観。要すれば商用再取り込み・タスク停止などのリカバリ調整。
- **B. 一時エラーだが最終ステータス「成功」**: `Lambda.TooManyRequestsException` 等がリトライで完了。ただし **DB再起動中など予期せぬタイミングの場合はデータ欠落の可能性**があるため、対象エンティティにデータが正しく格納されているか必ず確認。予期せぬ原因以外で成功 → **静観**。
- **C. 最終ステータス「失敗」**: 後続処理／Catch句・後続バッチで救済される仕組みか確認。復旧不可 → **要（連携元・顧客へ再連携を依頼、影響度 大寄り）**。

### 3-4. RDS の再起動・予期せぬ停止

- RDS停止中に実行された Lambda がエラーでも、リトライでDB復旧後に書き込み成功しているか確認。
- メトリクス確認: CPUUtilization（急騰）／FreeableMemory（枯渇）／DatabaseConnections（急増）。
- 判定: リトライ成功・整合性問題なし → **静観**。取りこぼしあり → **要**。

### 3-5. エラー以外で処理が行われない

- Lambda のトリガーを確認: SQS に滞留がないこと、EventBridge が有効であること。

## 4. エスカレーション・緊急扱いの条件

- 履歴テーブルで PK に対し current_flg=1・effective_month が同じレコードが2件以上（重複）→ 関係者・開発チームへエスカレーション。
- `strike-prd-ympf-sync-function` が連続発報 → **バッチがエラーを出し続け高額請求の恐れ**。速やかに管理者・関係部署へ報告し、原則本処理を停止して日付を挿入し再開（**影響度 大・要・緊急**）。

## 5. Lambda のリトライ設定 確認方法（自動復旧基盤を除く）

- SQSトリガー: 「デフォルトの可視性タイムアウト」とデッドレターキューの最大受信数（例: 15分毎・合計10回まで再実行）。
- Lambda: 「設定」→環境変数 `AWS_MAX_ATTEMPTS`。
- ステートマシン内の処理か: 「設定」→ステートマシンを確認。
