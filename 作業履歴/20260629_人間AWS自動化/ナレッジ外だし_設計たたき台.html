Phase2 検討資料

ナレッジ外だし 設計たたき台

取得したログの影響度と、人の目を通すべきかどうかをAI（Claude）に判定させるための、判定知識の持ち方・保存形式の検討資料です。

1目的・ゴール

Phase2の一項目として、取得済みのログ（ログ収集ツールがD列へ転記する内容）に対し、次の2点をClaudeに判定させたい。

  • 影響度（大／中／小）
  • 人の目を通すべきか（人レビューの要否）

これを安定して行うには、判定の拠りどころとなる知識を、Claudeが読みやすい形で外に出して保存・運用する必要がある。本資料では、外だしする知識の種類・保存形式・保存場所を整理する。決まっているのは「使うAIはClaude」のみで、他は未定。

前提: 本判定はPhase2以降のAI活用にあたる。ログの抽出そのものはAIを使わない（資格情報をAIに渡さない）方針だが、抽出済みテキストに対する判定は別の作業であり、AIの利用対象とする。

2全体像（判定の流れ）

知識を「常時渡すもの」と「必要時に検索して渡すもの」に分け、判定はルールとAIの二段で組む。

入力: 今回のアラーム ＋ 取得ログ（D列） ＋ [常時] 判定基準 ＋ ルール ＋ [検索] 類似の過去事例 数件 │ ▼ Claude が判定（構造化出力＝JSON） │ 影響度 / 人レビュー要否 / 理由 / 参照事例 ▼ ルールで機械的に上書き（安全側に補正） │ ▼ 結果をスプレッドシートへ転記

判定基準とルールは量が少なく安定しているため常時プロンプトへ載せる。過去の事象ログは増えていくため、当面は全件、増えたら類似分だけ検索して渡す。この3種類のナレッジは3章で詳しく述べる。

2-1各処理段階の担当（AI / プログラム）

この仕組みでAIが担うのは「判定」の1ステップだけで、それ以外はすべてプログラムが処理する。AIには抽出済みのテキストとナレッジしか渡さず、AWSにも資格情報にも触れない。

#処理段階担当補足
1対象シート（.xlsx）を読み、判定対象の行を抽出プログラムD列にログがあり判定が未実施の行
2アラーム名から環境・発生源種別を判定プログラムprd/stg、Lambda/StepFunctions など
3ナレッジ3ファイルの読み込みプログラムcases.jsonl / guidelines.md / rules.yaml
4類似の過去事例を検索して数件選ぶプログラムRAGの検索部分（将来導入。当面は全件渡すため検索なし）
5プロンプトの組み立て（ログ＋基準＋ルール＋事例）プログラム1リクエストにまとめる
6判定（影響度・人レビュー要否・理由）AI（Claude）構造化出力＝JSONで返す。AI処理はここだけ
7ルールで機械的に上書きプログラムrules.yaml の条件を適用（詳細は4章）
8結果をシート／中間ファイルへ書き出しプログラム 

RAGにあたるのは 4（類似事例の検索）で、これもプログラム処理。生成AIが動くのは 6 の判定だけで、そこ以外に「AIが考える」箇所は無い。

2-2検討した実現方法（なぜこの構成か）

「明確に決まる部分はルール、文脈判断が要る部分は生成AI」という今の形に至るまでに検討した主な選択肢を示す。

ア. 判定を何で行うか

選択肢長所短所判断
人手のみ（現状）確実件数が増えるほど手間がかかり、属人化する減らしたい対象
完全ルールベース
（正規表現・条件分岐だけ）		動作が読める・費用ほぼゼロ想定外の文言に弱い・ルールが膨れる・文脈を汲めない明確な規則だけ採用
自前の機械学習モデル大量処理に強い学習データの整備・運用が重く、小規模には過剰見送り
生成AI（Claude）で判定未知の文言にも対応・理由も出せる・少ない準備で始められる費用がかかる・出力がぶれ得る採用（ルールで補正）

→ 明確に決まる部分はルール、文脈判断が要る部分は生成AI、の二段構成が現実的。生成AIの弱点（出力のぶれ）は、ルールでの機械的な上書き（4章）で補う。

イ. AIに判断材料（知識）をどう持たせるか

選択肢長所短所判断
プロンプトに全部書き込む手軽直すたびにプログラム改修が要る・量が増えると管理不能一部のみ
知識を外だしして渡す（RAG的）知識と処理を分離・非エンジニアも直せる・増えたら検索で絞れる仕組みが少し増える採用
ファインチューニング
（Claudeへの追加学習）		毎回知識を渡さず軽い学習の手間・更新しづらい・判断根拠が見えにくい・少数事例に不向き見送り
長文にまるごと渡す単純事例が増えると費用も精度も悪化当面の全件渡しは近い形。増えたら検索へ

→ 知識を外に出して必要分だけ渡す形（RAG的）が、更新のしやすさと費用の両面で扱いやすい。

根拠と他事例（実在の技術・製品）

本構成の各要素は、確立された技術と実在する製品に基づく。

  • RAG（検索で知識を補ってから生成する）は確立された技術。2020年の論文「Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks」（Lewis ほか, NeurIPS 2020）で提案され、社内文書や最新情報にLLMを対応させる代表的な手法として定着している。今回の「事例を検索して渡す」部分がこれにあたる。
  • 監視・インシデント対応の分野では、AIに過去のインシデントやドキュメント（ランブック等）を検索させて一次調査・要約・トリアージを行う製品が実在する（例: incident.io の AI SRE、PagerDuty の SRE Agent／AIOps、Datadog のインシデント管理）。
  • カスタマーサポートの分野では、AIが問い合わせ内容をナレッジ（過去チケット・FAQ）と照らして分類・一次対応する製品が実在する（例: Zendesk AI の intelligent triage、Intercom の Fin）。分類結果を業務ルールに沿った振り分け（タグ付け・担当割り当て）につなぐ例もある。

※ 各製品の内部構成が本設計と同一だと主張するものではなく、「AI＋ナレッジ検索（RAG）＋ルール」という方向性が実務で使われている例として挙げている。
出典: RAG論文 (Lewis 2020) ／ incident.io AI SRE ／ PagerDuty AIOps ／ Datadog インシデント管理 ／ Zendesk AI

3外だしする3種類のナレッジ

役割の異なる3種類に分け、それぞれ読みやすい形式で別ファイルにする。スプレッドシートは行の追記が前提で列も固定のため、そのまま知識として読ませるには窮屈である。

3-1過去の事象ログ（判定事例）

「このアラーム・このログ内容に対して、こう判定した、その理由」という履歴。「前回と同件だから静観」を再現するための材料になる。

項目内容
用途類似事例の参照（過去と同じ事象なら同じ判定を再現）
保存する項目（案）アラーム名 / 発生源種別（Lambda・StepFunctions 等） / ログの特徴（キーワード・要約） / 判定結果（静観・要対応） / 影響度 / 人レビュー要否 / 理由 / 日時 / 参照リンク
保存形式JSONL（1件1行）。追記しやすく、件数管理・検索がしやすい。
元データ既存スプレッドシートの判定履歴を、この整形済みファイルへ吸い出す（変換する）。

検討点: 件数が少ないうちは全件をそのまま渡せる。増えたら「今回のアラームに近い数件だけ検索して渡す」方式に切り替える。まずは全件、増えたら検索、という段階移行が現実的。

3-2判定基準・考え方

事例が無い新しいパターンでも判断できるようにする指針。「影響度を大とみなす条件」「人の目を通すべき条件」を言葉で書く。

項目内容
用途未知のパターンでも判断できる、判断のよりどころ
保存形式Markdown（箇条書き）。プロンプトへ載せやすく、人が読んで直せる。
内容の例「本番（prd）で発生したエラーは影響度を一段上げる」「初めて見るエラー文言は人レビュー対象とする」

検討点: 量が少なく変化も緩やかなので、常時プロンプトへ載せる前提でよい。

3-3ルール（必ず適用する規則）

例外なく守る規則。ここはAIの判断に委ねず、決定的に適用したい部分。

項目内容
用途例外なく守る境界（安全側の担保）
保存形式YAML。機械的に読み取りやすいリスト形式。
内容の例「DLQ滞留は必ず要対応」「特定アラームは常に人レビュー」など、機械的にYES/NOが決まるもの

検討点: 機械的にYES/NOが決まる規則だけをここに置く。曖昧で判断が要るものは基準（3-2）としてClaudeに委ねる。

なぜこの3形式か（他形式との比較） 用途に合わせて分けた。事例は件数が増える構造データなので追記に強い JSONL、基準は人が読み書きする文章なので Markdown、ルールは機械が読む条件なので YAML。全部を1つのCSVやJSONにまとめる案もあるが、役割が混ざって直しにくくなるため分けた。将来、事例が数千件規模になり類似検索が重くなれば、意味の近さで探すベクトルDBも選択肢になる。

4ルールの適用方式の比較

ルール（3-3）をどう効かせるか。具体例「DLQ滞留は必ず要対応」で2案を比較する。

4-1（A）AIに渡して材料にする

ルール文をプロンプトに含め、Claudeに「これも考慮して判定して」と渡す。最終判定はClaudeが出す。

弱点 Claudeがルールを見落とす・解釈を誤る可能性が残る。DLQ滞留があるのに「静観」と返すことがまれに起こり得る。AIの出力は毎回まったく同じにはならないため。

4-2（B）AI判定を機械的に上書きおすすめ

Claudeにはいつも通り判定させる。そのあとプログラム側で「ログにDLQ滞留の兆候があるか」を単純に判定し、該当すれば結果を強制的に「要対応」へ書き換える。

利点 ルールは必ず効く。Claudeが何を返そうと、決められた規則は確実に守られる。

向き不向き: この方式に向くのは「YES/NOが機械的に決まるルール」だけ。曖昧で判断が要るものはClaudeに任せる。だから、はっきり決まる規則（3-3）と、判断の指針（3-2）を分けている。

観点(A) 材料にする(B) 機械的に上書き
ルールが必ず効くか効かないことがある確実に効く
実装プロンプトに追記するだけで簡単判定後の上書き処理が要る
向くルール曖昧なもの・多数のものYES/NOが機械的に決まるもの
安全性AIの出力に依存AIの出力に左右されない

結論: 確実に守りたい規則は（B）でプログラムが担保し、微妙な見極めはClaudeに委ねる役割分担とする。安全側に倒すため（B）を採る。

5Claudeへの渡し方

保存形式は「Claudeにどう渡すか」で決まる面が大きいため、渡し方も合わせて整理する。

  • 判定基準・ルール: 量が少なく安定 → 毎回プロンプトに固定で載せる。同じ前文を繰り返し使うため、プロンプトキャッシュで費用を抑えられる。
  • 過去の事象ログ: 増えていく → 当面は全件、増えたら類似検索して数件だけ渡す。
  • 出力: Claudeに構造化出力（JSON）で返させる。E列（対応要否）や新規列への転記、後段の月次集計にそのまま使える。

出力イメージ（案）:

{
  "影響度": "大",
  "人レビュー要否": "要",
  "理由": "本番のLambdaで初出のエラー文言。過去事例に一致なし。",
  "参照事例": []
}

6保存場所の比較（ファイル / DB）

ナレッジの置き場所を比較する。

観点ファイル（JSONL / Markdown / YAML）DB
追加インフラ不要。配布フォルダに同梱できるDBサーバ・その管理が要る
引き継ぎ容易性資材と一緒に渡せる（本ツールの方針に合う）接続情報・権限の受け渡しが要る
変更履歴テキストなので差分が見やすい。ナレッジだけ別途バージョン管理（Git・ドライブの版履歴）で残せる履歴管理は別途仕組みが要る
大量データ・複雑な検索件数が増えると重くなる得意
複数人の同時更新不得意得意
おすすめ: ファイル 本ツールはローカル手動実行・引き継ぎ容易性を重視しており、追加インフラの要らないファイルが合う。変更履歴はナレッジのファイルだけ別途バージョン管理で残せるため、ツール本体のGit管理は不要のまま、ナレッジだけ履歴を残す切り分けができる。

切り替えの目安: 事例が小〜中規模で同時編集も少ない今の段階はファイル。件数が数千件規模になり検索が重くなる、または複数人が同時に更新する段階になればDBを検討する。

7-2判定基準・考え方 … guidelines.md

事例に無いパターンでも判断できる指針。人が読んで直しやすいMarkdown。

# 判定基準・考え方

## 影響度の見極め
- 本番（prd）で発生したエラーは、同じ内容でも影響度を一段上げる。
- 利用者の操作が失敗する／データが欠落する・二重になるエラーは「大」。
- 自動リトライで回復済み、または一時的なタイムアウトのみは「小」。
- 短時間に同一エラーが多発している場合は影響度を上げる。
- 影響範囲が判断できない・情報が足りない場合は安全側に倒し「中」以上とする。

## 人レビューの要否
- 過去事例に一致せず、初めて見るエラー文言は「要」。
- 影響度が「大」のものは原則「要」。
- 既知の一時事象で、過去に静観と判断済みのものは「不要」。

7-3ルール … rules.yaml

例外なく守る規則。条件に当てはまれば、AIの判定を機械的に上書きする（4章の方式B）。

rules:
  - id: dlq-taiseki
    description: DLQ滞留は必ず要対応
    when:
      keywords_any: ["DLQ", "dead-letter", "デッドレター"]
    force:
      review: 要
      impact: 大

  - id: prd-sfn-failed
    description: 本番のStep Functions実行失敗は必ず人レビュー
    when:
      env: prd
      source: stepfunctions
      keywords_any: ["States.TaskFailed", "ExecutionFailed"]
    force:
      review: 要

  - id: prd-any-error
    description: 本番のエラーは影響度を「小」で確定させない
    when:
      env: prd
    force:
      review: 要

読み方: when＝この条件に当てはまるとき / force＝判定結果をこの値へ強制的に書き換える。env（環境）やsource（発生源）はアラーム名から判定する。

8起動方法・ツール構成

この判定の仕組みは、ログ収集ツールとは別の.pyとして用意する。収集ツールに組み込む案と分ける案を比較する。

観点① 収集ツールに組み込む② 別の.pyに分ける
収集ツールの「AI不使用」崩れる（同一処理内でClaudeを呼ぶ）保てる
判定に要るAWS権限収集側と混在不要（AWSに触れない）
担当分担まとめて1人が持つ前提AWS権限が無い人でも判定だけ動かせる
フェーズ境界抽出と判定が同居「抽出（AI不使用）／判定（AI使用）」の境界どおり
採用: ② 別の.pyに分ける 収集ツールはAWS資格情報とMFAを扱い、資格情報をAIに渡さないため意図的にAIを使わない作りにしている。判定は抽出済みテキスト（D列のログ）とナレッジを入力にClaudeへ渡すだけで、CloudWatchにもAWSにもアクセスしない。分けることで、収集ツールはこれまでどおりAIなしを保て、判定ツールはAWS権限・MFAを持たない人でも動かせる。

実行の流れ:

① ログ収集ツール .py … AI不使用・AWS/MFA あり │ （D列にログを転記） ▼ ② 判定ツール .py … AI使用（Claude）・AWS不要 （D列のログ ＋ ナレッジ → 影響度・人レビュー要否を判定）

運用は「収集ツール → 判定ツール」の順に続けて実行する。将来まとめて回したくなれば、外側の呼び出し用バッチ（1本）でつなぐこともできる。

9運用しながら精度を確認する（当面の進め方）

正解データを先に作り込むのではなく、まず仕組みを作り、運用チームが実際の目視判定と並行して使いながら、どれだけ外れるかをしばらく報告してもらう。小規模な案件では、これが現実に即していて手戻りも少ない。実運用の中で「人の目で確認しながら、ずれを改善へ回す」を行う形になる。

効果を出すための押さえどころ:

  • この期間はAIの判定を「提案」とし、正解は運用チームの目視とする。AIの結果は参考として別列に出し、人の判定と並べて見られるようにする。
  • 食い違った件を残す。下表の内容を簡単に記録しておくと、そのままナレッジへの反映材料になる。
  • 測るのは件数と不一致数だけで十分。難しい指標は要らない。「今週◯件中△件ずれた、内容はこれ」で、どこを直すかが見えてくる。

食い違いの記録（案）:

項目内容
アラーム / ログ対象の事象
AIの判定影響度・人レビュー要否
人の判定運用チームが下した正解

この記録を、種類に応じて3つのナレッジへ反映して精度を上げていく。

運用（AIは提案・人が確認） │ ▼ 食い違いを記録 │ ├─ 似た事象だった → cases.jsonl に事例を追加 ├─ 見極めの指針が足りない → guidelines.md を見直し └─ 必ず守るべき条件だった → rules.yaml にルール追加 │ ▼ 次回から判定が改善

ずれが十分に小さくなったと運用チームが判断できた段階で、AIの判定をどこまで自動採用するか（人レビューの範囲を狭めるか）を改めて検討する。

10ランニングコスト（特にClaude）

生成AIを使うため、判定を実行するたびにClaudeの利用料がかかる。ここでは費用の考え方と抑え方を整理する。（ログ抽出側は、ログ収集ツールで課金対象のLogs Insightsを使わないことで費用を抑えている。）

10-1費用の決まり方

  • Claudeは「入力トークン＋出力トークン」の従量課金。ざっくり「渡した文章量＋返ってきた文章量」に比例する。
  • 1件あたり費用 ＝ 入力（ログ＋基準＋ルール＋事例）＋ 出力（短いJSON）。出力は小さいので、費用の大半は入力側。
  • 月間費用 ≒ 1件あたり費用 × 月間件数。

10-2使用するモデルの選択

Claudeには、性能と価格の異なるモデルが複数ある。同じ処理でも、どのモデルを使うかで費用が数倍変わるため、費用に最も影響するのがこのモデル選びである。今回の判定（ログの内容を読んで分類する作業）は、文章を長く書かせる用途に比べれば軽い処理なので、まずは安いモデルで精度を確かめ、判定がずれるようなら上のモデルに変える、という順で選ぶと費用を抑えやすい。

モデル入力単価出力単価目安
Haiku（軽量）$1$5速い・安い。単純な判定向き
Sonnet（中位）$3$15速度と賢さのバランス
Opus（高性能）$5$25最も賢いが高い

単価は「100万トークンあたり」の金額（米ドル）。価格は改定されることがある。

10-3費用の概算

10-2は「単価」（1トークンあたりの値段）。ここでは、その単価に「1件で使うトークン数」と「件数」を掛けて、実際の金額を求める。

前提: 1件あたり 入力3,000トークン・出力150トークン、プロンプトキャッシュ未使用、1ドル≈150円。件数は 1日30件 × 20営業日 = 月600件と仮定。

計算の考え方（単価は「100万トークンあたり」なので、使うトークン数を掛けて金額にする）:

1件あたり費用 ＝ （入力トークン数 ÷ 100万 × 入力単価）＋（出力トークン数 ÷ 100万 × 出力単価） 月間費用 ＝ 1件あたり費用 × 月間件数 例）Haiku・1件: 入力 3,000 ÷ 1,000,000 × $1 ＝ $0.003 出力 150 ÷ 1,000,000 × $5 ＝ $0.00075 合計 ＝ 約 $0.004（約0.6円） 月600件 → $0.004 × 600 ＝ 約 $2.3（約340円）

この式に、各モデルの単価（10-2）を当てはめた結果が下表。

モデル1件あたり月600件
Haiku約 $0.004（約0.6円）約 $2.3（約340円）
Sonnet約 $0.011（約1.7円）約 $6.8（約1,000円）
Opus約 $0.019（約2.9円）約 $11（約1,700円）

あくまで概算。実際のトークン量と件数で変わる。現状の規模なら、月あたり数百円〜千数百円に収まる見込み。

10-4費用を抑える工夫

  • モデルの選択: 判定に足りる範囲でなるべく安いモデルを選ぶ（費用への影響が最も大きい）。9章の運用の中で、精度と費用のバランスを見て決める。
  • プロンプトキャッシュ: 毎回同じ基準・ルール・事例の入力部分をキャッシュし、2回目以降その部分を約1/10の価格で読める。
  • 事例を全件でなく検索で絞る（RAG）: 事例が増えたとき、入力トークンの膨張を抑える。
  • まとめ処理（バッチ）: 急がない集計なら、非同期でまとめて処理して単価を半額にする方法もある。

まとめ: 現状の規模では費用はごく小さく、モデル選択とプロンプトキャッシュで十分抑えられる見込み。
