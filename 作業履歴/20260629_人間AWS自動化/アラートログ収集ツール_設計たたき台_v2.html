Slack通知アラート ログ収集ツール 設計（たたき台 v2）

人手によるAWS運用作業の自動化 ― スプレッドシートのエラー行から CloudWatch ログを抽出し転記する（手動ファイル授受方式）

1. 目的 2. フロー 3. 検証結果 4. 4レイヤー検討 5. 手動操作と注意 6. ログ特定 7. 列レイアウト 8. 引き継ぎ

1. 目的・ゴール

これまで人手で行っていた「エラーのログ調査・転記」を半自動化する。最終的な対応要否の判断は人が行う。

  • やること: Slackアラートが集まるGoogleスプレッドシートの未処理行から、AWS CloudWatch のログを取得し、同じシートに転記する。
  • 実行形態: ローカルでの手動 Python 実行。Google側はプログラムから触らず、人手でファイルを授受する（.xlsxダウンロード → 処理 → 該当セルへ手動貼付）。
  • ゴール: 運用チームが「対応要否」を目視判断しやすい状態（ログ本文＋コンソールへのリンク）にする。
セキュリティ方針（最重要）: 本ツールは実行時に AIを使用しない。AWSアクセスキー・MFA等の資格情報をAIに渡さない／学習させないため、資格情報はローカルに閉じる。※開発段階でのAI活用は可。
方針転換の経緯（v2）: 当初は Python から Google API（サービスアカウント / OAuth / GAS Web App）で直接シートを読み書きする案を検討。しかし 匿名Web Appは組織が禁止・Google Cloud プロジェクトの利用も見送りとなったため、Google側はプログラムから触らず「人手でファイル授受」する方式に変更。これにより GCP・認証まわりが一切不要になる。
フェーズ上の位置づけ: Phase1=Slack収集（GAS稼働中）／Phase2＝本ツール（ログ抽出。AI不使用）／Phase3=AIが対応要否を判定／Phase4=月次集計。AIは本ツールの後工程（Phase3）で使うのが境界。

3. 検証結果（stg環境で動作確認）

2026-07-03、stg環境で試作版を実行し、「スプシ→CloudWatchログ取得→xlsxへ転記」まで、ひと通り動くことを確認した。

確認できたこと（stg）:
  • DLした .xlsx の未処理行(D空)の抽出
  • アラーム名 → describe_alarms → describe_metric_filters でロググループとフィルタを特定

4. 4レイヤーでの検討まとめ

「起動 / インプット / 情報取得 / アウトプット」の各レイヤーの採用方針。Google側（②④）は手動ファイル授受に変更。

① 起動
採用: ローカル手動 python 実行
MFA手入力と相性◎・キーをローカル完結
② インプット
採用: DLした .xlsx を読む
人手でDL → openpyxl で読取（GCP/API不要）
③ 情報取得
採用: アラーム→ログ特定
Insights不使用・命名規約に依存しない
④ アウトプット
採用: xlsxコピーへ転記→手動貼付
D=ログ / F=リンク を該当行へ

4-1① 起動レイヤー

案評価理由
ローカル手動 python 実行◎ 採用MFA手入力＋スイッチロールと相性◎。キーをローカル完結
OSタスクスケジューラ / cron 定期✗実行時のMFA手入力ができない
AWS側スケジュール(EventBridge+Lambda)✗「キーをローカル・手動MFA」方針と不整合
結論: 実行時にMFAを手入力する必要があるため、無人での自動実行はできない。「手動で起動し、その場でMFAを入力」が前提。

4-2② インプット ／ ④ アウトプット（Google連携＝手動ファイル授受）

案評価理由
手動DLした .xlsx を読む → 転記済みファイル出力 → 手動貼付◎ 採用GCP・認証が一切不要。組織審査もTLSプロキシ(Google側)も回避
Python から Google API 直接（SA / OAuth）✗GCP利用が見送りのため不採用
GAS Web App 経由✗匿名公開が組織で禁止のため不成立
利点と注意点: GCP・認証が不要で実装も単純になる一方、ダウンロードと貼付の手作業が増える。貼り戻し時は上書きに注意が必要（→ 5章）。

4-3③ 情報取得レイヤー（CloudWatch）

やりたいこと: アラーム名（B列）と発生時刻（A列）から、原因となった CloudWatch ログを取得する。詳しい辿り方は 6章。

決めること案評価理由
①どのロググループを見るかアラームから辿って特定（API）◎ 採用命名規約に依存しない
アラーム名の規則から推測△速いが規約変更に弱い（保険として併用可）
②ログをどう取るかget_log_events で前後N行取得◎ 採用スキャン課金なし（無料）
Logs Insights✗ 禁止スキャン課金が発生するため
認証: スイッチロール（assume_role）＋MFA。検証では正常動作を確認（アカウントごとにMFA要求＝3-2参照）。

5. 手動での操作と注意点

Google側を人手で授受するための手順と、安全に貼り戻すための注意点。

5-1手順

  1. 人手 対象スプレッドシートを開く → ファイル → ダウンロード → Microsoft Excel(.xlsx) を保存
  2. PC python run.py（入力＝DLした.xlsx）→ 実行年月のシートの未処理行(D空)を抽出 → MFA入力 → CloudWatchからログ取得
  3. PC 入力xlsxのコピーに D列（ログ）・F列（リンク）を転記したファイルを出力
  4. 人手 出力を見ながら、スプレッドシートの該当行の D列・F列だけを貼り付け

5-2貼り戻しの注意（重要）

丸ごと再アップロードはしない。 ダウンロード〜貼付の間も Phase1(GAS) が新しい行を追記し続けるため、出力ファイルを丸ごとアップロードするとその間に増えた行を上書きして消す恐れがある（元チケットの「download→upload不可」と同じ懸念）。→ 処理した行の D・F セルだけを貼る。
  • 行の対応: Phase1は末尾に追記のみ・並び替えなし。ダウンロード時点と貼付時点で既存行の行番号は一致する（間に手動で行挿入しない前提）。
  • 取り違え防止: 出力に照合キー（A列=日時 / B列=アラーム名）を併記し、貼付先の行が一致するか目視確認。
  • 増えた新規行: DL後に追記された行はD空のまま残り、次回実行の対象になる（取りこぼしなし）。

6. ログ特定のしくみ（CloudWatch）

B列のアラーム名さえあれば、課金対象の Logs Insights を使わず、AWSのAPIだけで対象ログまで辿れる。この方法は3章の検証で動作を確認している。

6-1入力列（A/B/C）の意味

列意味サンプル
Aタイムゾーン日付（UTC）2026-05-31 15:38:30
BALARM（CloudWatchアラーム名）strike-stg-app-log-errors-alarm
CTIME（JST＝A＋9h）2026-06-01 0:38:30

6-2ログ特定チェーン（Insights 不使用で成立）

B列アラーム名 duel-stg-delete-realtime-tran-work-errors-alarm └─▶ describe_alarms … アラームの Namespace / MetricName を取得 └─▶ describe_metric_filters … 「ロググループ名 + フィルタパターン」を取得 └─▶ filter_log_events … A列時刻の前後X分で該当イベントを特定 └─▶ get_log_events … 該当イベントの前後N行を取得（N可変） └─▶ コンソールのリンク生成 → F列へ
検証で確認: 上記チェーンで /aws/lambda/duel-stg-lambda-delete-realtime-tran-work（フィルタ "[ERROR]"）を特定し、前後N行を取得できた。GetLogEvents/FilterLogEvents はスキャン課金がかからない。

6-3エラー発生源による分岐（張り付ける内容が変わる）

発生源（Lambda / Step Functions / Fargate 等）によって、辿り方も張り付ける内容も変わる。判定キーはアラームのメトリクス種別。

メトリクス種別主な発生源取得経路D列（本文）に張り付ける内容状態
ログのメトリクスフィルタ（カスタム）Lambda / アプリ(Fargate・EC2)のログdescribe_metric_filters → ロググループ → filter / get_log_events該当ERROR＋前後N行検証済
AWS/Lambda（ネイティブ）LambdaFunctionName → /aws/lambda/{名前}該当ERROR＋前後N行簡易対応
AWS/StatesStep FunctionsStateMachineArn → 実行履歴API（失敗実行）失敗ステート＋ error / cause次段で対応
AWS/ECS 等Fargate / ECSタスクのロググループ該当ERROR＋前後N行要確認
実装方針: 発生源ごとに取得経路と出力の作りを切り替えられるようにする。Step Functions は前後N行が馴染まないため、失敗ステート＋error/cause＋実行画面リンクを主とする。どのアラーム種別があるかは事前に確認しておく。

7. 列レイアウト

出力先の列と役割。運用チームが目視で確認しやすいことを重視する。

列用途誰が書くか
A / B / C日付(UTC) / アラーム名 / 日時(JST)既存GAS(Phase1)が自動収集
DERRORS＝ログ本文（折返し表示・該当なし時は理由）本ツールが生成→人手で貼付
E対応要不要（色付け）人／Phase3のAI（ツールは触らない）
F（新規）「リンク」＝ログへのハイパーリンク本ツールが生成→人手で貼付
未転記の判定＝D列が空かどうか。該当ログが無い時もDに「該当なし＋理由」を記載し、常に非空にして二重抽出を防ぐ。ツールが出すD・Fを、人/Phase3がEの判断に使う。
処理対象行の方針: 当面は「D列が空の全行」を処理する。将来的に専用フラグ列を設けて処理対象だけを選別する案もあるが、取得したログは今後AIが各エラーの重要度（大／小）を判断するためのインプットになると考えているため、一旦は全行処理とする。

※ AWSのプロファイル（~/.aws/config）は各自のPC側にあり、この配布フォルダの外。README に「前提」として記載。

8-2config.yaml（各自の名称差を吸収）

プロファイル名など人によって違う設定は config.yaml に集約。コードは編集不要。

  • input_dir / output_dir（入出力フォルダ）／sheet_name（空=当月自動）
  • region ／ profiles（接頭辞→スイッチロール用プロファイル名）
  • lines_before / lines_after（取得行数N）／window_min（時間窓）
  • 列マッピング（入力A/B・出力 D=ログ / F=リンク）
  • ca_bundle（社内CA証明書のパス。空なら truststore）
社内CA: 社内TLS検査プロキシがあるため、AWS通信(boto3)では証明書対応が必要（truststore か 社内CA.pem）。Google通信は行わないためGoogle側のCA対応は不要。検証では truststore で疎通確認済み。
