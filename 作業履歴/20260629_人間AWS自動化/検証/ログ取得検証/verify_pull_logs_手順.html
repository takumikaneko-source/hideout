ログ収集ツール 運用手順

DLした .xlsx の未処理行（D列が空）→ CloudWatch ログ取得 → コピーの D/F 列へ転記 → 該当セルをスプレッドシートへ貼り付け

実行は手元（AWS認証は各自の環境・MFA） AI不使用 スクリプト: verify_pull_logs.py
目的 0. 事前準備 1. 入力の用意 2. AWSの前提 3. 設定値 4. 実行 5. 結果の見方 6. 貼り付け 7. 取得できなかった場合 8. 費用 9. 既知の制約

目的

ダウンロードした .xlsx の未処理行（D列が空）について、A列の日時・B列のアラーム名から CloudWatch のログを取得し、入力ファイルのコピーに対して D列（ログ本文）と F列（コンソールへのリンク）を書き込む。運用担当は、その D/F セルを本物のスプレッドシートの該当行へ貼り付ける。

出力は入力ファイルのコピー （入力名）_転記済.xlsx。同じシートの D列・F列に結果が入る。中身を確認し、該当行の D/F セルだけをスプレッドシートへ貼り付ける運用。
AWSの資格情報はスクリプトからは読み書きしません。boto3 が実行時に各自の ~/.aws を参照し、MFA はターミナルで入力を求められます。ログの判定にAIは使いません（AIを使うのは別ツール）。

0事前準備（1回だけ）

pip install boto3 openpyxl truststore

truststore は社内SSL検査（プロキシ）対策で、AWS通信の証明書エラーを回避します。効かない場合は設定の CA_BUNDLE に社内CA(.pem)のパスを指定してください。

1入力の用意（.xlsx ダウンロード）

  1. 対象スプレッドシートを開く → ファイル → ダウンロード → Microsoft Excel(.xlsx)
  2. 保存した .xlsx のパスを、スクリプトの INPUT_XLSX に設定
シートは実行年月(JST)から YYYY-MM を自動選択します。別の月を処理するなら SHEET_NAME="2026-06" のように明示してください。

2AWSの前提（スイッチロール・権限）

  • プロファイル: ~/.aws/config に、各環境のロールへスイッチするプロファイルがあること（role_arn / source_profile / mfa_serial）。
  • マッピング: スクリプトの PREFIX_TO_PROFILE に「アラーム接頭辞 → プロファイル名」を設定（例 "strike": "yzk-strike-stg"）。無い接頭辞は DEFAULT_PROFILE を使用。
  • リージョン: REGION（既定 ap-northeast-1）。
  • 必要なIAM権限（対象ロール）: cloudwatch:DescribeAlarms / logs:DescribeMetricFilters / logs:FilterLogEvents / logs:GetLogEvents。
MFAはアカウントごとに1回求められます。ワンタイムコードは使い回せないため、複数アカウントを処理する回では、プロンプトごとに新しいコードを入力してください。まず1アカウント・1プロファイルで MAX_ROWS=3 にして試すのが安全です。

3設定値（スクリプト上部）

INPUT_XLSX   = "（DLした.xlsxのパス）"
SHEET_NAME   = ""            # "" で当月自動 / 例 "2026-06"
OUTPUT_XLSX  = ""            # "" で 入力名+"_転記済.xlsx"
REGION       = "ap-northeast-1"
LINES_BEFORE = 100           # 該当行の前 何行
LINES_AFTER  = 100           # 該当行の後 何行
WINDOW_MIN   = 5             # A時刻の前後 何分を検索するか
MAX_ROWS     = 0             # 0=制限なし(運用) / 動作確認は 3 等
PREFIX_TO_PROFILE = { "strike": "yzk-strike-stg", ... }
DEFAULT_PROFILE   = ""       # 上記に無い時（""=既定資格情報）
CA_BUNDLE    = ""            # 社内CA(.pem) 任意
設定運用での目安
MAX_ROWS0（制限なし）で空き行をすべて処理。件数を絞って試すときだけ 3 等にする。
LINES_BEFORE / AFTER各100（前後で計200行）。長くて読みにくければ 50/50 程度へ。極端に長いログはセル上限で切られ、全文は F列リンクで見られる。
WINDOW_MIN±5分。「該当ログなし」が多い場合は 10〜15 に広げる。
PREFIX_TO_PROFILEアラーム名の先頭（例 strike-… の strike）と、その環境のプロファイル名を対応させる。

4実行

python verify_pull_logs.py
  • 対象プロファイルのロール引き受けで MFAコードの入力を求められます。
  • 進捗が行ごとに表示され、最後に （入力名）_転記済.xlsx が生成されます。
  • 末尾に集計が出ます: 転記OK / 該当ログなし / 取得できず・要確認 / 要再実行(D空)。

5結果の見方（コピーの D列の状態）

処理した行は、D列の中身で状態が分かります。

D列の中身意味対応
ログ本文が入っている OKロググループを特定し、前後N行を取得できた。F列にコンソールへのリンク。D/F を該当行へ貼り付け（6章）。
「【該当ログなし】…」ロググループは特定できたが、時間窓に一致するログがなかった。必要なら D を空にし WINDOW_MIN を広げて再取得。
「【取得できず】…」 要確認未対応の種類（Step Functions 等）・アラーム未検出・日時が不正 など。再実行しても変わらない。理由を確認し手動対応（7章）。
D列が空のままAWS通信・認証などの一時的なエラー。この行は処理されていない。もう一度実行すれば再取得します。
「D列が空＝未処理」です。だから一時エラーの行はあえて空のままにして、次回の実行で拾えるようにしています。

6スプレッドシートへ貼り付け

  1. 出力ファイル（…_転記済.xlsx）を開き、D列に結果が入った行を確認する。
  2. 本物のスプレッドシートの同じ行の D列・F列だけを、出力ファイルの値で更新する。
ファイルを丸ごとアップロード（差し替え）しないでください。ダウンロード〜貼り付けの間にも自動収集（Phase1）が新しい行を追記しているため、丸ごと差し替えると追記分が消えます。D/F セルの貼り付けだけにしてください。

7ログが取得できなかった場合

ケースD列対応
一時的なエラー（AWS通信・認証・MFA）空のままもう一度実行する。D空なので自動で再取得。
該当ログなし（窓内に一致なし）【該当ログなし】ログ未出力の可能性。必要なら D を空にし WINDOW_MIN を広げて再実行。
Step Functions のアラーム【取得できず】未対応の旨本ツールは未対応。実行画面を手動で確認。
アラームが見つからない【取得できず】未検出の旨アラーム名・対象アカウント/プロファイルの対応を確認。
A列の日時が不正【取得できず】日時解釈不可元の行の日時を確認。
原因を直してもう一度取得したいときは、その行のD列を空にして実行すると、未処理行として再取得します。

8費用（AWSからのログ取得）

このツールがAWSへ投げるのは、アラーム情報とロググループの特定、ログの取得だけです。使うAPIと費用は次のとおり。

API種類費用
describe_alarmsCloudWatchAPIリクエスト課金（$0.01 / 1,000件）だが、月100万件まで無料。本件数では実質無料。
describe_metric_filters
filter_log_events
get_log_events		CloudWatch Logsこれらの取得APIには、公式にリクエスト単価・GB単価の記載がない。Logsの課金は「取り込み・保存・Logs Insights（GBスキャン）」で構成される。
結論: AWSからのログ取得費用は、現状の件数では実質ほぼ0円。課金対象の Logs Insights を使わない設計にしていることが効いている。ログの取り込み・保存費用はアプリ側で既に発生している既存費用で、本ツールは読むだけなので追加は発生しない。

概算: 1日30件 × 20営業日 × 1行あたり最大4API ≈ 月2,400回。CloudWatchの無料枠（月100万リクエスト）に対して極小。
※ 判定AI（Claude）側の費用は別（「ナレッジ外だし 設計たたき台」10章）。

9既知の制約

  • 対応する種類: ログのメトリクスフィルタ型を主に自動対応。AWS/Lambda ネイティブも簡易対応。AWS/States（Step Functions）は未対応（実行履歴の取得は別途必要）。該当行は D に「【取得できず】」を記録するので、手動確認してください。
  • コンソールのリンク: 一般的な方式で生成。環境により開けない場合は、ロググループ／ストリーム名で手動遷移してください。
  • 複数該当: 窓内に複数のエラーがある場合、最初の該当を基点に前後N行を取得します（全件列挙は今後の拡張）。
