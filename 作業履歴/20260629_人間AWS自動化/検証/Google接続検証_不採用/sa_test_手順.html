サービスアカウント(SA) 接続テスト 手順書

「SA方式（各自認証なしでシート読み書き）が、この組織で使えるか」を手元で検証する

所要: 15分 2つの関門: ①SA鍵を作れるか ②SAでシート読み書きできるか 関連: sa_test_client.py
目的 0. 事前準備 1. プロジェクト/API 2. SA作成・鍵 3. シート共有 4. 実行 5. 結果の見方

目的

匿名Web Appが使えないと分かったため、代替のサービスアカウント方式が成立するかを確認する。確認すべき関門は2つ：

  1. SA鍵(JSON)を作成できるか（組織が サービスアカウント鍵の作成 を禁止していないか）
  2. そのSAでスプレッドシートを読み書きできるか
このテストは Googleのみ を扱い、AWSの資格情報は使いません。実行・鍵の取り扱いはすべてあなたの環境で完結します（結果は「成功／どこで詰まったか」だけ共有すれば十分）。
推奨: 本番ファイルではなく、テスト用の新規スプレッドシートで試す。

0事前準備

pip install gspread google-auth truststore

truststore は社内SSL検査(プロキシ)対策です（OSの証明書ストアを使用）。

1プロジェクトとAPI有効化

  1. Google Cloud Console に会社アカウントでアクセス
  2. プロジェクトを新規作成（例: sa-test）。作成できない場合は組織制限の可能性 → メモして報告
  3. APIとサービス → ライブラリ で Google Sheets API と Google Drive API を有効化

2サービスアカウント作成と鍵のダウンロード ★関門①

  1. IAMと管理 → サービスアカウント → サービスアカウントを作成（名前は任意、例 sheet-writer）。ロール付与は不要（シートアクセスは「共有」で行うため）。完了。
  2. 作成したSAを開く → キー → 鍵を追加 → 新しい鍵を作成 → JSON → ダウンロード
  3. ダウンロードしたJSONを sa_key.json にリネームし、sa_test_client.py と同じフォルダに置く
★関門①の判定: 「鍵を追加」で 「組織のポリシーによりブロックされました」「サービスアカウントキーの作成は無効」 等が出たら、組織がSA鍵を禁止しています＝SA方式はNG。その旨をメモして報告してください（→ 最終手段の構成へ）。

SAのメールアドレス（共有先）は xxxx@<project>.iam.gserviceaccount.com の形です（鍵JSONの client_email、またはSA一覧で確認）。

3テストシートをSAに共有 ★関門②の準備

  1. テスト用の空スプレッドシートを新規作成（https://sheets.new）
  2. 右上 共有 → SAのメール（...iam.gserviceaccount.com）を入力し 編集者 で共有
  3. そのシートのURLから ID（/d/ここ/edit）を控える
本番の共有ドライブで試す場合: 共有ドライブのメンバーにSAのメールを追加（コンテンツ管理者/投稿者）すれば、配下ファイルにアクセスできます。まずは上記の単独シートで関門②を判定するのが簡単です。

4スクリプト設定 & 実行

KEY_FILE   = "sa_key.json"
SHEET_ID   = "（控えたID）"
ALLOW_WRITE = True
CA_BUNDLE  = ""   # 社内CA(.pem)があればパス。無ければ truststore が使われる
python sa_test_client.py

5結果の見方

表示意味判定/対処
[1] 読み取り成功 ＋ [2] 書き込み結果: sa_write_okSAでの読み書き成功SA方式で本実装に進める
鍵作成が「ポリシーによりブロック」組織がSA鍵を禁止SA方式NG → 最終手段へ
シートを開けません … PermissionError(403)SAにシートが共有されていない手順3でSAメールに編集者共有 → 再実行
Sheets API has not been used/enabledAPI有効化漏れ手順1でSheets/Drive APIを有効化
SSL: CERTIFICATE_VERIFY_FAILED社内プロキシのCA未信頼pip install truststore、または CA_BUNDLE に社内CA(.pem)を指定
このテストで分かること: ①SA鍵が作れるか ②SAでシート読み書きできるか。両方OKなら、各自認証なしの引き継ぎやすい構成が確定します。
