アラート判定ツール 仕様・運用説明

2026-07-15 ／ 本人向け・全体像の整理用

要点（先に結論）

  • 2つのツールを順に流す仕組みです。①ログ収集ツール（AWSからエラーログを集めてExcelへ転記／AIは使わない）→ ②判定ツール（集めたログをAIが読み、対応要否・影響度を判定）。
  • AIの判定は参考情報で、最終判断は運用チームの目視（ExcelのE列）で行います。
  • 判定は案B（Opusオーケストレーション）で動作中。下位モデル（Haiku）が「過去事例・判定基準・システム構成」を読んで要約し、Opusが要約だけを見て最終判断します。
  • AIはサブスク（Claude Code / claude -p）経由で呼びます。APIキーは不要で、使用枠を消費します。
  • AI判定と運用判断の差分は別ファイル（AI判定_運用差分記録.xlsx）に蓄積し、ナレッジ改善に使います。
  • AWSの資格情報はAIに一切渡しません。 AWS接続を伴うログ収集はAIを使わず、判定ツールは「集めたログ本文」だけを読みます。

1. このツールは何か

1-1. 解決したいこと

CloudWatch のエラーアラートが Slack に通知され、スプレッドシートに集計されます。従来はその一件ごとに、担当者が CloudWatch のログを見に行き「対応が要るのか／静観でよいのか」を判断していました。このログ収集と一次判断を半自動化し、運用チームは要点を確認して最終判断に集中できるようにするのが目的です。

1-2. 全体の流れ

1-3. 2つのツールの役割分担

1枚のスプレッドシート（月ごとにシート、例 2026-07）を、両ツールが列を分担して埋めていきます。

3. ①ログ収集ツール

3-1. 何をするか

スプレッドシートの各行について、アラーム名からAWSプロファイル（スイッチロール先）を判定し、C列の時刻の前後（既定±5分）に出たエラーログを CloudWatch から取得して D列に転記します。あわせて元ログへのリンクを F列に書きます。入力ファイルは変更せず、コピーを作ってそこに書き込みます。

3-2. 処理する行の条件

次の条件をすべて満たす行だけを処理します。

  • A列（日時）とB列（アラーム名）がある
  • D列（ログ）が空 ＝ まだ収集していない行
よくある誤解 「一部の行しか処理されない」ときは、たいていD列が既に埋まっているのが原因です。再収集したい行は、入力ファイルのD列を空にしてから実行してください（G列など他の列は収集ツールの対象判定に関係しません）。

3-3. D列に転記されるログの形

後工程（AI判定・目視）が原因をすぐ読めるよう、発火した行（[ERROR]等）を先頭に再掲し、その後に前後の文脈を並べます。1行が長すぎる（巨大JSON等）場合は途中で切り、全文は F列のリンクから追えます。

■ 発火した行（アラームの検知パターンに一致）:
>>> [10:03:11] [ERROR] Task exception was never retrieved ...

── 前後の文脈（前30行 / 後40行。全文は F列リンク参照）──
    [10:03:10] post_cloud_taxi_s3_f1 : S3登録OK (HTTPStatusCode 200)
>>> [10:03:11] [ERROR] Task exception was never retrieved ...
    [10:03:12] 次サイクル処理を継続 ...

3-4. 主な設定（settings.py）

項目意味既定
INPUT_XLSXダウンロードした集計xlsxのパスINPUTフォルダ配下
PREFIX_TO_PROFILEアラーム名の接頭辞→AWSプロファイル対応表strike/duel/aegis/blitz/buster
SHEET_NAME対象シート（空＝当月を自動選択）空
WINDOW_MIN時刻の前後 何分を検索するか5
LINES_BEFORE / AFTER発火行の前後 何行を文脈に含めるか30 / 40
MAX_ROWS1回で処理する最大行数（0＝制限なし）0
補足 アラーム発火に対しログが見つからないことが多い場合は WINDOW_MIN を 10〜15 に広げると拾いやすくなります。

3-5. AWS接続とセキュリティ

このツールだけが AWS に接続します（スイッチロール・MFA）。AWSの資格情報（アクセスキー・MFA等）はAIに渡しません。 収集はAIを使わない機械的処理で、AIが関わるのは次の②判定ツールだけです。

4. ②判定ツール

4-1. 判定の流れ（1行ごと）

対象は「D列にログがあり（【取得できず】を除く）、G列が空」の行です。各行を次の順で処理します。

  1. 機械判定（rules.yaml）：明確なパターンはAIを使わず即断（費用ゼロ）。
  2. 類似事例検索（cases.jsonl）：過去の似た事例を機械的に探して材料にする。
  3. AI判定：ナレッジと類似事例をもとに、対応要否・影響度・理由・確信度を出す。
  4. （案Aのみ）確信度が低い場合：複数観点で再判定し多数決、割れたら上位モデルで最終判定。

4-2. 案B オーケストレーション（現在の方式）

「読む・要約する」作業は下位モデル（Haiku）に任せ、Opusは要約だけを見て判断に専念します。大きなナレッジ本文を毎回Opusに読ませない分、判断の質を保ちつつ役割を分けています。

4-3. 使うナレッジ4ファイル

ファイル役割使い方
rules.yaml機械判定ルール明確なパターンをAI前に即断（3件）
cases.jsonl過去事例（193件）類似事例を検索し判断材料に
guidelines.md判定基準静観/対応要の条件・影響度の目安
system_context.mdシステム背景処理フローと影響の伝播（上流起因・下流波及）

4-4. 出力される理由（I列）の体裁

理由は読みやすいよう、ブロックごとに空行で区切り、1文ごとに改行します。

【結論】ログ欠落のため断定できず静観とします。

【根拠】過去5事例は対応不要でした。
本件はログが無く両立します。
当該Lambdaはトリガー未発火の可能性が残ります。

【確認事項】Invocationsが0でないか目視で確認してください。

【前後関係】下流波及は限定的。
次サイクルで自然回復し得る。

5. 判定差分の記録（改善サイクル）

判定済みエクセルは実行のたびに入力から作り直される（洗い替え）ため、AI判定と運用判断の差分は別の永続ファイル AI判定_運用差分記録.xlsx に蓄積します。実行のたびに「日付＋アラーム名＋時刻」をキーにして更新追記（upsert）し、運用チームが記入した列は保持します。

列内容記入
日付 / アラーム名 / 時刻行を特定するキー自動
AI対応要否 / 影響度 / 確信度 / 判定日時判定ツールの出力自動（毎回更新）
運用対応要否運用チームの最終判断運用チーム
差分AIと運用が一致か差分か自動（数式）
差分理由・改善メモなぜ判断が分かれたか運用チーム
狙い 「運用対応要否」を入れると「差分」列が自動で 一致／差分 を表示します。差分の理由を見直して rules.yaml や cases.jsonl を育てることで、判定精度を継続的に上げていきます。

6. 運用手順（通し）

  1. 準備（初回）：各ツールで pip install -r requirements.txt。判定ツールは claude にログイン。
  2. ①収集：集計xlsxをダウンロードして INPUT に置き、settings.py を確認して collect_logs.py を実行（AWS接続あり）。
  3. ②判定：まず MAX_ROWS を小さくして judge_logs.py を試し、問題なければ広げる。
  4. ③最終判断：判定済みxlsxの G〜K を参考に、運用チームが E列に最終判断を記入。
  5. ④改善：差分記録xlsxに運用判断と理由を記入し、差分をナレッジ更新に反映。
注意 実行時は対象のExcelファイルを閉じてください。開いたままだと書き込みに失敗します。

7. 費用と性能の目安

案B・サブスク経由での実測は、5行で約$2相当・1行あたり約105秒でした。全312行なら約$120・約9時間が目安です。

なぜ費用がかかるか 費用の主因はログの長さよりも、claude -p の1回ごとに再作成される約2万トークンの土台コンテキストです。案Bは1行につきAIを4回呼ぶため、呼び出し回数が費用を大きく左右します。軽くしたい場合は案A（1行1回）へ切り替えると概ね1/4程度になります（精度は案Bより落ちます）。

8. セキュリティ上の決まり

  • AWSの資格情報（アクセスキー・MFA等）はAIに渡さない・読み込ませない。
  • AWS接続を伴うログ収集はAIを使わない。判定ツールが読むのは「収集済みのログ本文」だけ。
  • APIキーはソースに書かず、環境変数のみ（サブスク運用ならキー自体が不要）。
  • 資料・記録には個人名を残さず、「運用チーム／関係者」等に置き換える。

9. 配布セットの構成

他の人へ渡せるよう、必要なものを1フォルダに集約しています。相対パスはフォルダ基準で解決されるため、フォルダごと移動しても動きます。

配布セット/
├─ はじめにお読みください.md
├─ 1_ログ収集ツール/  (collect_logs.py, settings.py, requirements.txt, 手順書.html, INPUT/, OUTPUT/)
└─ 2_判定ツール/      (judge_logs.py, settings.py, requirements.txt, README.md, ナレッジ/, OUTPUT/)

各ツールは settings.py だけを編集すれば動き、本体（*.py）の変更は不要です。
