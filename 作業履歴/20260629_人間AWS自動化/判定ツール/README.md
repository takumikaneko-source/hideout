# アラートログ判定ツール（Phase3）

ログ収集ツールが転記したエラーログ（D列）を読み取り、AI（Claude）で
**対応要否（要/静観/不要）** と **影響度（大/中/小）** を判定して G列以降に書き込むツール。

判定はあくまで参考情報であり、**最終判断（E列）は運用チームの目視**で行う。

## 2つの切替軸

このツールは settings.py で「**どこ経由でAIを呼ぶか（バックエンド）**」と
「**判定の進め方（方式）**」を切り替えられる。処理を支えるナレッジ4ファイルは共通。

### バックエンド（`BACKEND`）
| 値 | 内容 |
|----|------|
| `"claude_code"`（既定・実運用） | Claude Code のサブスクを `claude -p` 経由で使う（APIキー不要・追加費用なし・使用枠を消費） |
| `"api"` | Anthropic API（従量課金・要 ANTHROPIC_API_KEY） |

> サブスク版の前提: `npm install -g @anthropic-ai/claude-code` で `claude` を導入し、一度ログイン（IDEで既にログイン済みなら共有される）。
> 実行は `claude` がPATHにある通常のターミナル（PowerShell等）で行う。動作確認済み。

### 判定方式（`JUDGE_MODE`）
| 値 | 内容 |
|----|------|
| `"orchestration"`（既定・案B） | 下位モデル（Haiku）が照合・判定基準の読取・システム構成の読取を**要約** → Opusがその要約で最終判定。理由が厚い／使用枠は約4倍 |
| `"pipeline"`（案A） | 機械照合 → 類似検索 → Sonnet判定 →（確信度低なら）多観点多数決 → Opus再判定。使用枠が軽い |

### 案A（pipeline）の流れ
```
1. rules.yaml と機械照合（AI不使用・費用ゼロ）→ 一致で確定
2. cases.jsonl から類似の過去事例を検索（機械・上位5件）
3. Sonnet判定（判定基準＋システム背景＋ログ＋類似事例）
4. 確信度「低」→ 3観点（業務影響/復旧・再処理/事例照合）で多数決
5. それでも割れる → Opusで最終判定
```
確信度が高い件は 3 の1回で終わるため、費用が増えるのは曖昧な件だけ。

### 案B（orchestration）の流れ
```
1. rules.yaml 機械照合 → 一致で確定
2. 類似事例検索（機械）
3. 作業者(Haiku/Sonnet) ×3: 過去事例の要約 / 判定基準の該当部抽出 / システム構成の所見
4. 指揮官(Opus): 3つの要約とログをもとに最終判定（大きなファイル本文は読まない）
```
Opusは判断だけに使うため全部Opusより安いが、1件で複数回モデルを呼ぶ。

## セットアップ（初回のみ）

```powershell
pip install -r requirements.txt
```

**API を使う場合**（`BACKEND = "api"`）: APIキーを環境変数で渡す（**settings.py やソースに書かないこと**）。
```powershell
$env:ANTHROPIC_API_KEY = "sk-ant-..."
```

**サブスクを使う場合**（`BACKEND = "claude_code"`）: APIキーは不要。
Claude Code を導入し、一度 `claude` でログイン（サブスク認証）しておく。
判定ツールが内部で `claude -p` を呼んで判定する。使用枠を消費する点に注意（下記）。

## 実行

```powershell
python judge_logs.py
```

設定（入力ファイル・モデル・件数上限など）は [settings.py](settings.py) だけを編集する。

### まずお試し（APIキー不要・費用ゼロ）

settings.py で `DRY_RUN = True` にすると、AIを呼ばずに
機械判定と類似事例検索だけを実行して対象件数を確認できる。

## 出力

入力ファイルの**コピー**（`<入力名>_判定済.xlsx`）に書き込む。入力ファイルは変更しない。

| 列 | 内容 |
|----|------|
| G | AI対応要否（要/静観/不要） |
| H | AI影響度（大/中/小） |
| I | AI判定理由（前後関係の所見つき） |
| J | AI確信度（高/中/低。機械判定は「確定」） |
| K | 判定方式（機械:ルール名 / AI:モデル名 / AI指揮:… など、どの経路で判定したか） |

E列（運用チームの対応要不要）には**触らない**。

## 費用・使用枠

### API バックエンドの場合（従量課金）
- 判定基準・システム背景（約1.1万トークン）はプロンプトキャッシュに載るため、2件目以降は約1/10
- 案A: 1件 **約1〜2円**（曖昧な件のみ3〜5倍）。案B: 1件 **約5〜15円**（Opusは判断のみ）
- 実行後にトークン使用量・概算コストが表示されるので実測で確認できる

### サブスク（claude_code）バックエンドの場合
- **追加のお金はかからない**（プラン定額に含まれる）。代わりに**使用枠**を消費する
- 案Aは1件1回、**案Bは1件で複数回（作業3回＋判断1回）呼ぶため使用枠を数倍消費**する
- 日次20件程度なら上位プランの枠内に収まる見込み。過去分の一括判定や無人連続運用は
  枠に当たりやすいので分散実行を推奨（実際の上限はご契約プランで要確認）

## 安全のための決まりごと

- **AWSには一切アクセスしない**（このツールはxlsx・ナレッジファイル・Claude（API か Claude Code）しか触らない）
- APIキーは環境変数からのみ読む（サブスク運用ならAPIキー自体が不要）
- 入力ファイルは非改変（コピーに書き込み）。出力先が入力と同じ場合は中止
- 1回の実行は既定200行まで（settings.MAX_ROWS）。無限ループなし
- 判定済みの行（G列に値あり）は再実行時にスキップ（冪等。settings.RE_JUDGE で変更可）

## ナレッジの更新

判定精度は `../ナレッジ外だし/` の4ファイルで決まる。

- 運用チームの目視結果とAI判定がズレたら、guidelines.md にケースを追記する
- 「答えが決まっている」パターンが見つかったら rules.yaml に昇格させる
  （機械判定になり、そのパターンは費用ゼロ・確実になる）
- Backlogを再取得したら `../ナレッジ外だし/build_cases.py` で cases.jsonl を再生成する

## うまくいかないとき

| 症状 | 対処 |
|------|------|
| `ANTHROPIC_API_KEY が設定されていません` | 上記の環境変数を設定。お試しは DRY_RUN=True |
| `APIに接続できません` | 社内プロキシ配下では `pip install truststore` が必要 |
| `保存できません` / `出力先に書き込めません` | 出力xlsxをExcelで開いていたら閉じる |
| `anthropic パッケージが古く…` | `pip install -U anthropic` で更新 |
| `G1 に想定外の見出し…` | シートのG〜K列が別用途で使われている。列割り当てを要相談 |
| `レート制限` | 時間をおいて再実行（判定済み行はスキップされるので続きから進む） |
| キャッシュが効いていない旨の注意 | 想定外の状態。ナレッジファイルを実行中に変更していないか確認 |
