Slack通知アラート ログ収集ツール 設計（たたき台）

人手によるAWS運用作業の自動化 ― スプレッドシートのエラー行から CloudWatch ログを抽出し転記する

1. 目的 2. 4レイヤー検討 3. フロー 4. 実現方法 5. 検証・論点 ★ 6. ログ特定 7. 列レイアウト 8. 引き継ぎ

1. 目的・ゴール

これまで人手で行っていた「エラーのログ調査・転記」を半自動化する。最終的な対応要否の判断は人が行う。

  • やること: Slackアラートが集まるGoogleスプレッドシートの未処理行から、AWS CloudWatch のログを取得し、同じシートに転記する。
  • 実行形態: ローカルでの手動 Python 実行（無人の自動実行はしない）。
  • ゴール: 運用チームが「対応要否」を目視判断しやすい状態（ログ本文＋コンソールへのリンク）にする。
セキュリティ方針（最重要）: 本ツールは実行時に AIを使用しない。AWSアクセスキー・MFA等の資格情報をAIに渡さない／学習させないため、資格情報はローカルに閉じる。※開発段階でのAI活用は可。
フェーズ上の位置づけ: Phase1=Slack収集（GAS稼働中）／Phase2＝本ツール（ログ抽出。AI不使用）／Phase3=AIが対応要否を判定／Phase4=月次集計。AIは本ツールの後工程（Phase3）で使うのが境界。

結論: MFA手入力が必須要件である以上、無人の自動スケジュールは原理的に不可。「手動起動＋実行時MFA」が大前提。

2-2② インプット ／ ④ アウトプット（Google連携）

Google側の読み書き方式は 4章（実現方法の候補）・5章（検証結果）で詳述。要点のみ：匿名GAS Web App＝検証で不可／サービスアカウント直Sheets API＝本命（要GCP・各自認証なし）／OAuth直＝予備。決定後、入力（未転記行の取得）と出力（D=ログ・F=リンクの書込）は同じ方式で行う。

2-3③ 情報取得レイヤー（CloudWatch）

やりたいこと: アラーム名（B列）と発生時刻（A列）を手がかりに、原因となった CloudWatch のログを取得する。

採用する方法: ①アラームから「そのアラームが見張っているロググループ」をAPIで特定 → ②その時間帯の該当ログを無料のログ取得APIで前後N行取得。課金される Logs Insights は使わない。具体的な辿り方は 6章「ログ特定のしくみ」 を参照。

4. 実現方法の候補（Google連携の手段）

「Python（ツール）がスプレッドシートを自動で読み書きする手段」の候補を、前提知識なしで説明します。プログラムが自動で読み書きするには「正式な入口（API）」と「身分証明（認証）」が要り、そのやり方が複数あります。

4-1候補①：GAS Web App　検証の結果：不可

既存のGAS（Google上で動くプログラム）に「外から呼べる窓口」を付け、PCのツールがそれを呼んでシートを読み書きしてもらう案。ツール側の設定が最も少なくて済むが、「誰でも呼べる窓口（匿名公開）」を社内ルールが禁止しており、呼ぶとログイン画面に弾かれた（→ 5章）。

4-2候補②：サービスアカウント　本命・要確認

「プログラム専用の利用者（ロボットのアカウント）」を1つ作り、そのアカウントにシートを共有する。ツールは鍵ファイルを使って人の代わりに読み書きする。利用者ごとのログインが不要で引き継ぎが楽。ただしGoogle Cloudプロジェクトと鍵ファイルの作成が必要で、社内が許すか要確認（→ 5章）。

4-3候補③：OAuthユーザー認証　予備

ツールをあなた自身（ログインした人）として動かす案。初回だけブラウザで「許可」を押す。使う人ごとに初回操作が要る。GCPプロジェクトも必要。

4-4候補④：AWS中継　最終手段

どうしてもGoogleへ直接入れない場合に、AWS側を“受け渡し場所”にして連携する案（既存GASは外向き通信は可）。しくみが複雑になるため最終手段。

AWS側をGASに載せない理由（参考）: AWSの資格情報・MFAをGoogleクラウド側に置けず、対話的なMFA入力もGASのモデルに乗らないため。＝AWS処理はローカルPythonに置く。

5. 検証結果と論点（会議で決めたいこと）★

Google連携の方式を実機検証した結果、当初案（匿名GAS Web App）は組織制限で不可と判明。方式の再決定が必要。本資料の中心論点。

5-1検証で判明した社内制約

項目結果と影響
社内TLS検査プロキシ自己署名のルート証明書を挟むため、AWS/Google両方の通信で証明書対応が必須（truststore か 社内CA(.pem) 指定）。
GAS匿名Web App公開設定で「全員」が選べず「会社内の全員」まで＝匿名公開不可。当初案（GCP不要）は不成立。
用語解説：社内TLS検査プロキシとは
会社のネットワークには、社外とのHTTPS（暗号化）通信をいったん開封して中身を検査する機器があり、その際、相手（Google/AWS）の証明書を会社独自の証明書（自己署名のルート証明書）に差し替えて通します。
  • ブラウザは普通に見える：会社の証明書をIT部門が事前に「信頼済み」として配布しているため。
  • Python（boto3/requests）はエラーになる：会社の証明書を知らず「信頼できない証明書」と判断する（実際テストで CERTIFICATE_VERIFY_FAILED が発生）。
  • 対策＝プログラムにも会社の証明書を信頼させる：① truststore でOS（Windows）の証明書ストアを使う／② 会社のルート証明書(.pem)を環境変数 AWS_CA_BUNDLE/REQUESTS_CA_BUNDLE 等で指定。
  • AWS/Google両方で必要：CloudWatch（AWS）もスプレッドシート（Google）も同じ社内プロキシを通るため。検証を切る(verify=False)のではなく「正しく信頼させる」のが本筋。

※既存Phase1のGASが動くのは「Google内部で時間トリガー実行（外部からの着信なし）」だから。本ツールは外部（ローカルPC）からGoogleへ入るため、匿名着信を禁止する組織ポリシーに当たる。

5-2残った候補と方針

案GCP各自の認証状態
① 管理者へ匿名公開を申請不要不要不可見込み
② サービスアカウント（本命）要不要可否を確認したい
③ OAuthユーザー要要予備
④ AWS中継不要不要最終手段
用語解説：GCP（Google Cloud Platform）と「プロジェクト」とは
GCPはGoogleのクラウドサービスの総称。その中の「プロジェクト」は、プログラムからGoogleの機能(API)を使うための“管理の箱”です。プログラムがGoogleのAPIを使うには、この箱の中で ①プロジェクトを作る → ②使うAPI(Sheets API)を有効化 → ③認証情報を発行、という準備を行います。
作成時の最初のページ（GCPコンソール）:
・トップ: https://console.cloud.google.com/
・プロジェクト作成: https://console.cloud.google.com/projectcreate
候補GCPで何をするか（使い方）GCPが要る理由
② サービスアカウントプロジェクト作成 → Sheets API有効化 → サービスアカウントを作り鍵(JSON)を発行 → 鍵をツールに同梱・シートをSAに共有SAの作成と鍵の発行がGCPプロジェクト内の機能だから
③ OAuthユーザープロジェクト作成 → Sheets API有効化 → 同意画面設定 → OAuthクライアントIDを発行 → 初回ブラウザ認証「このアプリです」と名乗るOAuthクライアントIDの発行がGCP内だから
①（GAS Web App）・④（AWS中継）がGCP不要なのは、①はGASがGoogleユーザーとして既存の許可で動くため、④はGoogleの認証情報を使わずAWS経由にするため。

6. ログ特定のしくみ（CloudWatch・技術の核）

B列のアラーム名さえあれば、課金対象の Logs Insights を使わず、AWSのAPIだけで対象ログまで辿れる。これが本ツールの技術的な核。

6-1入力列（A/B/C）の意味

列意味サンプル
Aタイムゾーン日付（UTC）2026-05-31 15:38:30
BALARM（CloudWatchアラーム名）strike-prd-app-log-errors-alarm
CTIME（JST＝A＋9h）2026-06-01 0:38:30

6-2ログ特定チェーン（Insights 不使用で成立）

B列アラーム名 strike-prd-app-log-errors-alarm └─▶ describe_alarms … アラームの Namespace / MetricName を取得 └─▶ describe_metric_filters … 「ロググループ名 + フィルタパターン」を取得 └─▶ filter_log_events … A列時刻の前後X分で該当イベントを特定 └─▶ get_log_events … 該当イベントの前後N行を取得（N可変） └─▶ コンソールのリンク生成 → F列へ
ポイント: GetLogEvents/FilterLogEvents はスキャン課金の対象外。よって「Logs Insights（課金）は使わない」要件を満たしつつ、アラームと同じ条件で原因ログを特定できる。

6-3エラー発生源による分岐（張り付ける内容が変わる）

発生源（Lambda / Step Functions / Fargate 等）によって、辿り方もシートに張り付ける内容も変わる。判定キーはアラームのメトリクス種別（Namespace / Dimensions）。describe_alarms の結果で判定し切り替える。

メトリクス種別主な発生源取得経路D列（本文）に張り付ける内容F列（リンク）
ログのメトリクスフィルタ（カスタム）
※サンプルの *-log-errors-alarm はこれと推定		Lambda / アプリ(Fargate・EC2)のログdescribe_metric_filters → ロググループ → filter / get_log_events該当ERROR全件＋前後N行Logsストリーム
AWS/Lambda（Errors 等のネイティブ）LambdaDimension FunctionName → /aws/lambda/{名前} → filter / get_log_events該当ERROR全件＋前後N行Logsストリーム
AWS/States（ExecutionsFailed 等）Step FunctionsDimension StateMachineArn → Step Functions API（ListExecutions=FAILED → GetExecutionHistory）失敗ステート＋ error / cause（前後N行より有用）SF実行画面
AWS/ECS 等Fargate / ECSタスクのロググループ → filter / get_log_events該当ERROR＋前後N行Logsストリーム
設計への反映: 取得経路・出力フォーマット・リンク種別を発生源ごとの差し替え可能な部品（ストラテジ）として実装する。特に Step Functions は「行ベースの前後N行」が馴染まないため、失敗ステートと error / cause ＋ 実行画面リンクを主とする。※「前後N行」は Lambda / アプリのような行ベースのログで有効な概念。
注意（チェーン成立の前提）: 採用チェーン（describe_alarms → describe_metric_filters → …）が成立するのは「ログのメトリクスフィルタ」型のみ。AWS/States 等のネイティブメトリクスのアラームでは metric_filters が存在せず、上表の別経路が必要。実際のアラーム種別の網羅確認が前提。

7. 列レイアウト

出力先の列と役割。運用チームの目視確認しやすさを最優先。

列用途誰が書くか
A / B / C日付(UTC) / アラーム名 / 日時(JST)既存GAS(Phase1)が自動収集
DERRORS＝ログ本文（該当なし時は理由を記載）本ツール
E対応要不要（色付け）人／Phase3のAI（ツールは触らない）
F（新規）ログストリームへのリンク本ツール
未転記の判定＝D列が空かどうか。該当ログが無い時もDに「該当なし＋理由」を必ず書き、常に非空にして二重抽出を防ぐ。ツールが出すD・Fを見て、人/Phase3がEを判断する流れ。
