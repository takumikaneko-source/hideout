3. バッチ仕様 (6 本)

処理概要 / 起動タイミング / 取得元 / 連携先 / 連携先アカウントでの利用処理 / 処理内容 / 実行 SQL を記載。

バッチ ①解析後日報 JSON 取り込み (get_json_from_strike)

処理概要strike アカウントから配信された 解析後日報 JSON ファイル を、DWH (Redshift) に取り込む処理です。1 ファイル＝乗務員 1 日分の運行記録に相当します。
起動タイミングS3 バケット s3-link-dwh-json に JSON が配置されると SQS イベントが発火し、本バッチが即時起動します。
取得元S3: taxi-cloud-buster-${EnvType}-s3-link-dwh-json (strike が配置)
連携先Redshift Serverless (database: taxicloud / schema: public) の各業務テーブル
処理内容
  1. SQS 経由で受け取った S3 オブジェクトキーから対象 JSON を取得
  2. ファイル名形式を検証し、data_key(レコード一意キー) を生成
  3. JSON 内容を Pydantic でスキーマバリデーション
  4. JSON 内のセクション (共通 / 運行 / 入出庫 / 営業 / 決済 / ETC / イベント / 評価) ごとに分割し、対応する 約 30 テーブル に登録
  5. バリデーション失敗時は 日報ワークテーブル (nippou_wk) にエラーフラグ付きで JSON 文字列のまま退避
  6. 取り込み成功後、元の S3 JSON を削除
実行 SQL テーブル単位に「一時 CSV を S3 にアップ → Redshift COPY で一括投入」する方式 
COPY "taxicloud"."public"."unko"
  (data_key, cdimg_id, isdegi_flg, ..., insert_datetime, insert_user_id, insert_business_id)
FROM 's3://taxi-cloud-buster-${EnvType}-s3-link-dwh-json/tmp/{data_key}_{table}_{YYYYMMDDHHMMSS}.csv'
IAM_ROLE 'arn:aws:iam::...:role/...redshift_role'
FORMAT CSV
REGION AS 'ap-northeast-1';

※ Redshift Data API による非同期発行。execute_id を DynamoDB に登録し、バッチ⑤で完了判定。

エラー時挙動スキーマ違反 / 重複キー違反などは nippou_wk に退避。DB エラー時は SQS リドライブで再試行。SNS 経由で Backlog 起票

バッチ ②DWH→duel 連携 (export_dwh_data)

処理概要DWH に蓄積された日報・運行データを、動態 BI アカウント (duel) の S3 に CSV で配信します。
起動タイミング10 分毎 (cron(0/10 * * * ? *))
取得元Redshift Serverless taxicloud DB (対象テーブルは DynamoDB params で動的管理)
連携先S3: taxi-cloud-duel-${EnvType}-s3-link-dwh (duel アカウント所有)
連携先アカウントでの利用処理
duel アカウント側にこの CSV を取り込む Lambda は本リポジトリ内に存在しない。
duel 側で S3 (s3-link-dwh) を購読しているのは LinkDailyJsonRule (EventBridge) だが、フィルタが object.key.suffix: .json (JSON 限定) のため、本バッチが出力する CSV はトリガにならない。
CSV は duel の ECS マートタスク (duel-prd-ecs-bi) または BI 側集計処理が S3 から直接読み込み、Aurora MySQL マートを更新していると推測される。
※本リポジトリの SAM テンプレートからは確認不可、ECS タスク定義／Step Functions ワークフロー側を要確認。
処理内容
  1. パラメータマスタから連携対象テーブル一覧を取得
  2. 各テーブルの「前回成功時刻＜更新／登録時刻＜今回最新時刻」の差分行のみを抽出
  3. 結果を Redshift UNLOAD で duel 側 S3 に CSV 出力
  4. 今回最新時刻を batch-status テーブルに保存し、次回の差分起点とする
実行 SQL 
-- 最新時刻取得
SELECT MAX(insert_datetime), MAX(update_datetime) FROM <table>;

-- UNLOAD で差分抽出 & 出力
UNLOAD ('SELECT * FROM <table>
  WHERE ((update_datetime IS NULL AND insert_datetime <= ''<最新>'')
      OR (update_datetime IS NOT NULL AND update_datetime <= ''<最新>''))
    AND ((update_datetime IS NULL AND insert_datetime > ''<前回>'')
      OR (update_datetime IS NOT NULL AND update_datetime > ''<前回>''))')
TO 's3://taxi-cloud-duel-${EnvType}-s3-link-dwh/<table>/<YYYYMMDDHHMMSS>.csv'
IAM_ROLE 'arn:aws:iam::...:role/...redshift_role'
FORMAT AS CSV HEADER ALLOWOVERWRITE PARALLEL OFF;

バッチ ③DWH→aegis 連携 (export_dwh_redshift_data)

処理概要DWH の特定テーブル群を、マスタメンテアカウント (aegis) の S3 に CSV で配信します。
起動タイミング10 分毎 (cron(0/10 * * * ? *)) ※「暫定」コメントあり
取得元Redshift Serverless taxicloud DB (対象テーブルは DynamoDB params で動的管理)
連携先S3: taxi-cloud-aegis-${EnvType}-s3-link-dwh (aegis アカウント所有)
連携先アカウントでの利用処理
aegis 側で import_dwh_redshift_data Lambda が SQS 経由で起動 → CSV を Aurora MySQL に Upsert で投入する。
パス: backend/aegis/data_link_batch/functions/import_dwh_redshift_data/import_dwh_redshift_data.py
INSERT INTO <table> (col1, col2, ...) VALUES (:col1, :col2, ...) AS new
ON DUPLICATE KEY UPDATE col1 = VALUES(col1), col2 = VALUES(col2), ...
取り込み完了後、aegis 側で元 S3 CSV を delete。aegis 側 Aurora MySQL の運行データテーブル群が、軌跡表示・乗務員日報など aegis 画面の参照ソースとなる。
処理内容②と同じ「差分 UNLOAD 方式」。配信先 S3 のみ aegis 側になる点が違い。連携対象テーブルもパラメータマスタで個別管理される。
実行 SQL (buster 側)バッチ ② と同一構造の UNLOAD (出力先 S3 バケットのみ差し替え)

バッチ ④マスタ取り込み (sync_master_data)

処理概要aegis が管理する 乗務員・営業所・車両・顧客などのマスタ CSV を、DWH に取り込みます。
起動タイミングaegis が連携用 S3 に CSV 配置 → S3 イベント → 本アカウントの SQS (data_collect) 着信時
取得元S3 (CSV) ※ SQS メッセージ内の bucket/key から動的解決
連携先Redshift Serverless taxicloud DB の対応マスタテーブル (jomuin_mst / eigyosyo_mst / syaryo_mst / kokyaku_mst / client_mst 等 約 20 テーブル)
処理内容
  1. SQS から S3 オブジェクトキーを取得
  2. キー先頭ディレクトリ名 = テーブル名として採用 (例: jomuin_mst/202605211200.csv → jomuin_mst)
  3. Redshift COPY で当該テーブルへ全件投入
  4. クエリ ID を DynamoDB に登録し、バッチ⑤で完了判定
実行 SQL 
COPY "taxicloud"."public"."<table_name>"
FROM 's3://<bucket>/<key>'
IAM_ROLE 'arn:aws:iam::...:role/...redshift_role'
FORMAT AS CSV
DELIMITER ','
REGION AS 'ap-northeast-1';

バッチ ⑤Redshift クエリ実行ステータス管理 (redshift_execute_status_manager)

処理概要他バッチ (①②③④) が 非同期発行した Redshift クエリの完了状態を巡回監視 し、成功したクエリの一時 S3 ファイルを掃除します。
起動タイミング5 分毎 (cron(0/5 * * * ? *))
取得元DynamoDB redshift-execute-status の全 execute_id
連携先 (操作対象)DynamoDB redshift-execute-status / S3 一時 CSV
処理内容
  1. DynamoDB から実行中クエリ ID 一覧を取得
  2. Redshift Data API の describe_statement で照会
  3. ステータス別処理:
    • FINISHED → 元 S3 ファイル削除 + DynamoDB レコード削除
    • FAILED → エラーログ出力 (バッチ⑥ で再実行のため DynamoDB はそのまま)
    • ABORTED → ログ出力
    • 実行中等 → DynamoDB の status / update_datetime を更新
実行 SQL(SQL 発行なし / Redshift Data API describe_statement のみ)

バッチ ⑥失敗クエリ再実行 (retry_redshift_query)

処理概要バッチ⑤で FAILED 判定された Redshift クエリを改めて発行し直し、一時的な障害によるデータ欠損を防ぎます。
起動タイミング1 時間毎 (cron(0/60 * * * ? *)) ※「暫定」コメントあり
取得元DynamoDB redshift-execute-status (status = 'FAILED' のレコード)
連携先Redshift Serverless (再 COPY / UNLOAD 発行)
処理内容
  1. DynamoDB を FAILED で絞り込みスキャン
  2. 各レコードの query 列に保存された SQL を再発行
  3. 新しい execute_id で DynamoDB を上書き、旧レコードは削除
実行 SQL元レコードに保存された SQL をそのまま再実行 (内容はバッチ①④由来の COPY または ②③ 由来の UNLOAD)

4-3. 連携データがどのようにレコード化されるか

4-3-1. 解析後日報 JSON (LT27 → strike → buster)

1 つの JSON ファイル (= 乗務員 1 日分) が、Redshift 上で 1 件の common レコード + 関連 TX (トランザクション) 数十件 に展開されます。

{1乗務分のJSON}
   ├─ common(キー部)             → common テーブル (1 レコード / data_key)
   ├─ unko(運行サマリ)            → unko テーブル (1 レコード / data_key)
   ├─ hyoka(評価サマリ)           → hyoka テーブル (1 レコード / data_key)
   ├─ eigyo[](営業 1 回 × N)      → eigyo テーブル (N レコード / data_key + renban)
   ├─ nyusyu[](入庫/出庫)         → nyusyu テーブル
   ├─ kessai[](決済明細 × N)      → kessai テーブル
   ├─ etc[](ETC 通過 × N)         → etc / etc_gate テーブル
   └─ event[](イベント × N)       → event テーブル

4-3-2. マスタ CSV (aegis → buster)

ファイル名先頭ディレクトリと同名のテーブルに COPY で全件登録。例: eigyosyo_mst/20260521_120000.csv → テーブル eigyosyo_mst 全件投入。

4-5. データ保持サイクル (TTL／削除実装の事実)

4-5-1. Redshift テーブル

観点結果 (ソースコードからの事実)
DDL 上の TTL 指定なし。Redshift 自体に TTL 機能はなく、CREATE TABLE に保持期間指定もない。全 56 テーブルで TTL 系の宣言は存在しない。
物理削除を行うバッチ本リポジトリ内に存在しない。grep -rn "DELETE FROM" backend/buster の結果 0 件。BATCH_ID_* 定数にも削除系バッチ定義なし。
論理削除フラグマスタ系のみ delete_flg decimal(1,0) NOT NULL DEFAULT 0 を持つ。aegis 側マスタの delete 操作に追従して 1 に立てる運用 (本バッチでは物理削除しない)。
VACUUM/再編成自動実行コードなし (Redshift Serverless の自動メンテナンスに依存)。
結論 (コード上の事実)DWH 上の業務レコードは無期限保持。コード上の物理削除実装は存在しないため、運用側で別途 SQL を発行する以外にレコードは消えない。
※ 「過去 2 年分」は運用方針／設計書由来であり、本リポジトリのコードからは実装根拠が確認できない。

  • 業務 TX (トランザクション) の物理削除は本リポジトリでは未実装。Redshift Serverless はストレージ課金 (RMS) があるため、保持方針 (例: 2 年で削除) を持つ場合は別途運用バッチか手動 SQL の整備が必要。
  • DynamoDB の運用メタは件数が膨らみやすい (特に batch-log)。本来は TTL を設定すべきだが現状未設定 → 別途運用検討推奨。

5. 料金削減施策 (取り込み停止案)

本章の目的: 料金削減のため、DWH への「取り込み処理」のトリガーを停止し、Redshift への書き込みを止めることで料金削減ができないかを検証する。あわせて、停止中のデータ欠損リスクと復旧可否を整理する。

5-2. 料金削減の有効性

5-2-1. 削減対象 (大きい順)

コスト要素削減効果補足
Redshift Serverless 計算 (RPU 秒)大COPY / UNLOAD クエリが発生しなくなるため、最大の削減源
Redshift Serverless ストレージ (RMS)無し既存データを保持する限り課金は継続
Lambda 実行小取り込み Lambda 実行回数が 0
S3 ストレージ逆増取り込み停止中、JSON が滞留して増加
SQS リクエスト小配送は継続、Lambda 側で受信されない
DynamoDBほぼゼロ取り込みが無ければ書き込み発生せず
CloudWatch Logs小ログ出力が減る

5-2-2. 評価

  • 有効である: 料金は Redshift Serverless の RPU 秒に大きく依存する設計で、COPY/UNLOAD を止めれば計算課金は劇的に減少する。
  • 「① のみ停止」では効果が限定的。書き出し ②③ も 10 分毎に動いているため、これらも併せて停止しないと Redshift の起動時間は残る。
  • ストレージ料金は減らない。完全削減には DWH 自体の停止 (workgroup 削除 → snapshot 化) が必要。
  • 業務影響: 取り込みを止める＝BI 画面のデータが鮮度を失う。

5-2-3. 推奨される停止パターン

A. 軽量停止①④ のみ中取り込み停止のみ。BI には既存データ範囲で配信が続く
B. 全バッチ停止①②③④⑤⑥ すべて大DWH への/からの全ての動きが停止
C. アカウント全停止上記 + Redshift workgroup 削除最大 (ストレージ含む)復旧に時間と手順が必要

5-3-3. 結論

完全欠損のリスクは極めて低い。停止中も生データは前段の S3 に残るか、aegis の Aurora マスタに正がある。

ただし 2 つの条件付きリスク がある:

  1. s3-link-dwh-json バケットの ライフサイクルルール。本リポジトリ内にバケット定義およびライフサイクル定義は存在せず、別 IaC で管理されているため、設定値は AWS コンソール要確認。取り込み停止期間より短い期限が設定されていれば JSON が消失する。
  2. SQS link_dwh_data の MessageRetentionPeriod。本リポジトリ内にキュー定義も無く、設定値は要確認。停止期間が保持期間を超えると再開時に自動取り込みが走らない (手動キック必要)。

上記 2 点を確認・調整しておけば、取り込み停止 → 再開で自然復旧する運用が成立する。

5-4. 停止／再開の操作チェックリスト

停止時

  • Lambda buster-${EnvType}-get-json-from-strike の Event Source Mapping (SQS link_dwh_data) を Disable
  • Lambda buster-${EnvType}-sync-master-data-function の Event Source Mapping (SQS data_collect) を Disable
  • (任意) export-dwh-data-function の Schedule を Disable
  • (任意) export-dwh-redshift-data-function の Schedule を Disable
  • (任意) redshift-execute-status-manager-function の Schedule を Disable
  • (任意) retry-redshift-query-function の Schedule を Disable
  • 事前確認: s3-link-dwh-json バケットの Lifecycle 設定
  • 事前確認: SQS link_dwh_data / data_collect の MessageRetentionPeriod (推奨 14 日)
  • 事前確認: aegis 側マスタ連携の S3 容量
  • Backlog/SNS 経由のアラート抑止を検討

再開時

  • Event Source Mapping / Schedule を順に Enable
  • SQS にメッセージが残っていれば自動で取り込みが進む
  • SQS 期限切れ時は S3 内 JSON 一覧から再キック
  • 取り込み完了後 tmp/ 配下に残存 CSV があれば手動清掃
  • aegis に対しマスタ全件再連携を依頼
